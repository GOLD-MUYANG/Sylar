# 项目健康度审计记录

## 目标

识别当前源码中除“功能尚未扩展”外的功能重复、构建可达性和实际不可用问题；本次只审计，不修改业务代码。

## 排查命令与原因

- `rg --files -g '!build/**' ...`：建立源码、测试、配置和文档的完整清单，确认审计范围。
- `sed -n '1,280p' CMakeLists.txt`：核对哪些实现和测试真正进入构建图，发现“源码存在但无法验证/交付”的问题。
- `git status --short` 与 `git log --oneline -12`：区分用户当前未提交改动和最近已完成的功能，避免误判。
- `rg -n -i 'TODO|FIXME|...'`：定位显式未完成标记、断言型失败路径与潜在不可用实现。

后续会继续检查声明与实现、关键调用链、测试覆盖和可执行验证；每条结论都会标注证据等级。

## 继续排查的命令与结果

- `cmake --build build -j2`：确认当前 CMake 图可完整构建；构建通过，但输出显示多数基础测试目标仍被注释。
- `./bin/test_http_request_options`、`./bin/test_http_client`、`./bin/test_http_circuit_breaker`、`./bin/test_http_concurrency_limiter`：验证已接入且不依赖监听 socket 的 HTTP 新增能力；均通过。
- `./bin/test_http_load_balance_client`、`./bin/test_http_connection`：检查端到端 HTTP 路径；本环境禁止创建 loopback socket（`socket ... errno=1 Operation not permitted`），因此失败不能归因于业务实现。该环境限制也在既有记录中已知。
- `./bin/sylar -p -c bin/conf`：无副作用地检查文档中的启动参数；日志显示配置路径被解析为 `<仓库>/bin/bin/conf`，目录不存在但加载函数没有报错，证实该命令会静默跳过实际配置。
- `./bin/sylar -p -c conf`：对照验证正确的相对路径；日志显示 `<仓库>/bin/conf`，并加载到 `worker.yml`，证实路径问题不在配置文件本身。
- `rg -n -C ... 'onLoad|onServerReady|onConnect' ...`：检查模块生命周期声明是否有调用点；只发现参数解析与卸载的调用，未发现加载/服务就绪/连接生命周期接入。
- `rg -n 'SSL_CTX_set_verify|SSL_set_tlsext_host_name|...' sylar`：检查 HTTPS 客户端身份校验；没有结果，而 `SSLSocket::connect` 只调用 `SSL_connect`。
- `nl -ba ...`：按行号核对 WorkerGroup、连接池、Application、Env、CMake 和测试断言的具体控制流，作为结论定位依据。
- `ctest --test-dir build --output-on-failure`：检查是否具备标准测试发现入口；结果为 `No tests were found`，因为项目没有 `enable_testing()` 或 `add_test()`。

## 已确认的重点

- `WorkerGroup` 没有把 `batch_size` 传给 `FiberSemaphore`，其默认并发数为 0；首次 `schedule()` 就会等待且没有任务能归还信号量。
- 模块动态库可以被加载，但测试模块的 `onLoad()` 在实际 `-p` 运行中没有输出；框架没有在加载、服务 ready/up、连接建立/关闭时触发生命周期回调。
- `HttpConnectionPool::Create()` 对非法 URI 只记录日志，却继续解引用空 URI 指针，存在确定的空指针崩溃路径。
- README 对同一套 `WorkerGroup`/`WorkerManager` 连续用“worker 模块”“线程池”“封装线程池”三处描述，容易让读者误以为存在三套能力；实际只有这一套，且当前 `WorkerGroup` 的准入路径不可用。
