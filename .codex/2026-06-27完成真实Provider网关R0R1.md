# 2026-06-27 完成真实 Provider 网关 R0/R1

## 背景

根据 `文档/真实提供商网关.md`，本次先完成：

- R0：TLS 客户端配置补齐与 HTTP 尝试阶段元数据。
- R1：`GatewayChatRequest/Response`、`ProviderCandidate`、逻辑模型路由和 provider-aware executor。

## 排查与验证命令

1. `sed -n '1,260p' 文档/真实提供商网关.md`
   - 原因：确认 R0/R1 的验收边界，避免把 R2 及之后的真实适配器、错误分类和预算模型提前混入。

2. `sed -n '1,420p' sylar/http/http_connection.cc`、`sed -n '1,360p' sylar/http/http_client.h`、`sed -n '1,320p' sylar/socket.h`
   - 原因：确认现有 HTTP 请求路径、连接池路径、`SSLSocket::ClientOptions` 的边界，判断 TLS 配置和尝试阶段元数据应该落在哪一层。

3. `sed -n '1,620p' modules/ai_gateway/ai_gateway_upstream.cc`、`sed -n '1,360p' modules/ai_gateway/ai_gateway_servlet.cc`
   - 原因：确认当前 Mock 网关仍通过统一 path/body 的 `HttpLoadBalanceClient` 工作，R1 新能力需要独立 provider-aware 组件，避免破坏 G0-G5。

4. `cmake -S . -B build`
   - 原因：新增测试目标后刷新构建系统；初次失败暴露 `real_provider_gateway.cc` 尚未实现，符合 TDD 的 RED 阶段。

5. `cmake --build build --target test_http_attempt_outcome`、`cmake --build build --target test_ai_gateway_real_provider`
   - 原因：先编译 R0/R1 新增契约测试，确认缺失点集中在 HTTP 尝试元数据和真实 provider 路由/执行器。

6. `./bin/test_http_attempt_outcome`
   - 原因：验证连接失败标记为 `not_sent`、读超时标记为 `response_timeout_after_send`。普通沙箱禁止创建 AF_INET socket，因此使用提升权限重跑；该测试只用本地回环，不访问外网。

7. `ctest --test-dir build -R 'http_attempt_outcome|ai_gateway_real_provider|ai_gateway_protocol|ai_gateway_servlet|ai_gateway_load_balance|http_request_options|http_client|tls_client_options' --output-on-failure`
   - 原因：回归 R0/R1 新测试和现有网关协议、Servlet、负载均衡测试，确认 Mock G0-G5 不被新真实 provider 结构破坏。

8. `./bin/test_http_request_options`、`./bin/test_http_client`
   - 原因：这两个目标没有注册进当前 CTest 正则命中的集合，直接运行二进制验证 HTTP timeout/client 行为。

9. `./bin/test_http_connection`
   - 原因：本次修改了 `HttpConnection`，按项目验证规则运行该测试；它依赖外网访问 `www.baidu.com` 和 `www.sylar.top`，因此结果受网络环境影响。

10. `cmake --build build`
    - 原因：最终全量构建验证 `libsylar.so`、测试目标、`ai_gateway_module` 和主程序都能通过编译链接。

## 结果

- 新增 `HttpAttemptOutcome` 和 `HttpAttemptPhase`，HTTP 结果能区分未发送、发送前失败、partial/未知、发送后读超时、收到响应。
- `HttpRequestOptions` 增加 TLS server name、CA file/path 和 verify peer 配置，HTTPS 连接池创建连接时传递到 `SSLSocket`。
- 新增真实 provider 网关基础类型、逻辑模型路由和 provider-aware executor。候选保留各自 adapter、base URL、chat path、upstream model 和 API Key 环境变量名。
- 新增 `tests/test_http_attempt_outcome.cc` 和 `tests/test_ai_gateway_real_provider.cc`，并接入 CMake/CTest。
- 相关测试、外网 `test_http_connection` 和全量构建均通过。
