# 2026-06-27 实现 R4 真实 Provider 错误分类

## 背景

根据 `文档/真实提供商网关.md` 的 R4，补齐 Provider 错误分类与非幂等重试安全。目标是不启用真实外网 Provider，只在离线契约层覆盖供应商状态码、发送阶段元数据和脱敏错误对象。

## 排查与验证命令

1. `sed -n '1,260p' 文档/真实提供商网关.md`
   - 原因：先确认 R4 的任务边界，避免把真实 Provider 启用、R5 limiter/breaker 或 R6 HTTPS 夹具混进本次实现。
   - 结论：R4 只要求 Provider 错误分类与非幂等重试安全，真实 Provider 访问仍默认关闭。

2. `sed -n '1,320p' modules/ai_gateway/real_provider_gateway.cc`、`sed -n '1,320p' modules/ai_gateway/ai_provider_adapter.cc`
   - 原因：确认真实 Provider executor 和 OpenAI-compatible adapter 的职责边界。
   - 结论：候选选择在 `ProviderAttemptExecutor`，Provider HTTP 请求/响应转换在 `AiProviderAdapter`，错误分类应放在 executor 所属的真实 Provider 层，adapter 只复用分类结果生成脱敏错误响应。

3. `rg -n "R4|real_provider|AiProvider|Provider|adapter|真实" modules/ai_gateway tests examples bin/conf CMakeLists.txt 文档/真实提供商网关.md`
   - 原因：确认当前测试目标和真实 Provider 相关源码都已接入构建。
   - 结论：`test_ai_gateway_real_provider` 已在 CMake 中启用，适合作为 R4 的离线契约测试。

4. `cmake --build build --target test_ai_gateway_real_provider`
   - 原因：写入 R4 失败测试后，先确认测试本身能编译。
   - 结论：测试目标编译通过。

5. `./bin/test_ai_gateway_real_provider`
   - 原因：验证 RED，确认新增测试能暴露 R4 缺口。
   - 结论：初次运行失败，主要表现为 429 没有切换候选，401 直接透出 provider 原始响应体，包含测试 Key 和用户消息。

6. `cmake --build build --target test_ai_gateway_real_provider`、`./bin/test_ai_gateway_real_provider`
   - 原因：实现分类与脱敏错误响应后，验证 R4 目标测试。
   - 结论：目标编译和单测均通过。

7. `cmake --build build`
   - 原因：确认新增 public contract 不只在单测目标内可用，模块库也能链接。
   - 结论：完整构建通过，包含 `libai_gateway_module.so`。

8. `ctest --test-dir build -L unit --output-on-failure`
   - 原因：跑 unit label 做回归检查。
   - 结论：16 个 unit 中 14 个通过，`http_attempt_outcome` 和 `ai_gateway_load_balance` 失败都出现 `socket(2, 1, 0) errno=1 errstr=Operation not permitted`，属于当前沙箱禁止本地 socket/listen 的环境限制；本次 R4 目标 `ai_gateway_real_provider` 通过。

9. `ctest --test-dir build -R ai_gateway_real_provider --output-on-failure`
   - 原因：在全量 unit 受 socket 沙箱影响时，单独确认 R4 离线目标。
   - 结论：`ai_gateway_real_provider` 通过。
