# 2026-06-25 实现 AI 网关 G4

## 排查目标

实现 `文档/AI模型调用网关方案.md` 中的 G4：把 AI Gateway 接入已有 HTTP 客户端 limiter、breaker、健康检查和总尝试预算，保持网关模块、上游装配、Servlet 响应语义的职责边界清楚。

## 使用过的命令和原因

1. `sed -n '1,260p' 文档/AI模型调用网关方案.md`
   - 原因：先确认 G4 的精确范围，避免把 G5 `/internal/status` 或真实 Provider 工作混进来。

2. `rg -n "G4|熔断|breaker|failover|deadline|attempt|retry|internal/status|status" 文档/AI模型调用网关方案.md modules/ai_gateway tests examples CMakeLists.txt`
   - 原因：快速定位 G4 文档要求、现有 AI 网关代码、测试和构建目标。

3. `sed -n '1,320p' modules/ai_gateway/ai_gateway_upstream.cc`、`sed -n '1,300p' modules/ai_gateway/ai_gateway_module.cc`
   - 原因：确认网关当前只创建了 G3 负载均衡客户端，尚未把 limiter/breaker 配置传给 `HttpLoadBalanceClient`。

4. `sed -n '1,320p' sylar/http/http_load_balance_client.h`、`sed -n '1,420p' sylar/http/http_load_balance_client.cc`
   - 原因：确认底层已经有 `HttpConcurrencyLimiter`、`HttpCircuitBreaker` 和 `checkHealth()`，G4 应复用它们而不是在网关层重写状态机。

5. `cmake --build build --target test_ai_gateway_load_balance test_ai_gateway_servlet`
   - 原因：TDD 红灯验证。新增测试先引用 `AiGatewayUpstreamOptions` 和 `HttpRetryOptions::max_total_attempts`，确认生产代码尚未实现这些边界时会失败。

6. `./bin/test_ai_gateway_load_balance`
   - 原因：运行目标测试。沙箱内出现 `socket(...) errno=1 Operation not permitted`，说明该测试需要本地 loopback socket 权限。

7. `./bin/test_ai_gateway_load_balance`（提权）
   - 原因：排除沙箱限制后验证真实本地回环逻辑。第一次提权后发现使用普通 `std::thread` 发起并发请求会和当前 IOManager/hook 测试夹具不匹配，表现为连接超时。

8. 修改测试为 endpoint QPS 准入路径后重新运行：
   - `cmake --build build --target test_ai_gateway_load_balance test_ai_gateway_servlet ai_gateway_module`
   - `./bin/test_ai_gateway_servlet`
   - `./bin/test_ai_gateway_load_balance`（提权）
   - 原因：验证最终 G4 行为，包括限流切换、全部限流返回 429、熔断跳过、健康检查和总尝试预算。

## 结论

- G4 的主体缺口不是底层治理能力，而是 AI 网关没有把 YAML 配置装配到已有 `HttpLoadBalanceClient`。
- `RATE_LIMITED` 原先会被 Servlet 统一映射成 502；现在改为 429，并隐藏底层内部错误文案。
- `max_total_attempts` 原先只存在于文档草图；现在进入 `HttpRetryOptions`，并由 `HttpLoadBalanceClient` 在跨 endpoint 尝试前统一扣减。
- 本地 loopback 测试在沙箱内会失败，需要提权运行；这属于环境限制，不是 G4 逻辑失败。
