# 2026-06-26 解释网关请求成功但 recv 失败日志

## 问题

用户发起 AI Gateway 请求后，业务响应正常，但日志里仍出现：

`recv http request fail, errno=11 errstr=Resource temporarily unavailable`

## 排查命令

1. `rg -n "recv http request fail|handleClient|EAGAIN|Resource temporarily unavailable|keepalive|setRecvTimeout|getRequest" sylar/http sylar/socket.* sylar/streams -S`
   - 原因：定位日志字符串和可能相关的 keepalive / recv 调用。
   - 结论：日志来自 `sylar/http/http_server.cc:32`；命令中 `sylar/streams` 路径不存在，是无影响的搜索噪声。

2. `nl -ba sylar/http/http_server.cc | sed -n '1,90p'`
   - 原因：查看日志打印所在的控制流。
   - 结论：`HttpServer::handleClient()` 在 keepalive 循环里调用 `session->recvRequest()`；返回空时统一打印 WARN 并断开连接。

3. `nl -ba sylar/http/http_session.cc | sed -n '1,180p'`
   - 原因：确认 `recvRequest()` 什么时候返回空。
   - 结论：底层 `read()` 返回 `<= 0` 时会 close 并返回 `nullptr`。

4. `nl -ba sylar/stream.cc sylar/socket_stream.cc 2>/dev/null | sed -n '1,220p'`
   - 原因：追踪 `HttpSession::read()` 最终如何到 socket recv。
   - 结论：`SocketStream::read()` 调用 `m_socket->recv()`；无数据、关闭或超时都会向上表现为读失败。

5. `nl -ba examples/ai_gateway_conf/server.yml | sed -n '1,80p'`
   - 原因：确认 demo 网关 HTTP Server 是否启用 keepalive。
   - 结论：配置中 `keepalive: 1` 且 `timeout: 1000`。

6. `nl -ba sylar/application.cc | sed -n '330,370p'`
   - 原因：确认 `server.yml` 的 keepalive 配置如何传给 HTTP Server。
   - 结论：`Application` 用 `i.keepalive` 构造 `HttpServer`。

## 结论

这条 WARN 不是 `/v1/chat/completions` 业务请求失败。业务响应发出去以后，由于 server 开启 keepalive，`HttpServer::handleClient()` 会继续在同一连接上等待下一条 HTTP 请求。客户端没有继续发送数据，或连接进入空闲直到超时，底层 `recv` 返回失败并带出 `errno=11 EAGAIN`，于是当前代码统一打印 `recv http request fail`。

## 可选处理

- 只做演示：可以忽略，它是连接收尾/空闲 keepalive 的噪声。
- 想减少日志：把 `examples/ai_gateway_conf/server.yml` 的 `keepalive` 改成 `0`，或请求时加 `Connection: close`。
- 想从框架层优化：在 `HttpServer::handleClient()` 区分正常客户端关闭、keepalive 超时、解析失败，不要把所有 `recvRequest() == nullptr` 都打成 WARN。
