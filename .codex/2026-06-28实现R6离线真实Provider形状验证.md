# 2026-06-28 实现 R6 离线真实 Provider 形状验证

## 排查目标

核实并实现 `文档/真实提供商网关.md` 的 R6：用本地录制响应或假 HTTPS 服务验证真实 Provider 的 OpenAI-compatible 非流式响应形状，不依赖外网、真实 API Key 或计费账户。

## 使用过的命令与原因

1. `sed -n '1,260p' 文档/真实提供商网关.md`
   - 先读取任务书，确认 R6 的范围是“离线真实 Provider 形状验证”，并保持真实 Provider 默认关闭。

2. `sed -n '1,1040p' tests/test_ai_gateway_real_provider.cc`
   - 查看现有 R2/R3/R4/R5 回归，确认 R6 应继续放在真实 Provider adapter/executor 测试里，而不是新建外网测试。

3. `sed -n '1,320p' modules/ai_gateway/ai_provider_adapter.cc`
   - 查看 OpenAI-compatible adapter 的解析逻辑，定位它当前只检查 `choices[0].message.content`，对 `id/object/created/model/role/finish_reason/usage` 等真实响应形状校验过宽。

4. `cmake --build build --target test_ai_gateway_real_provider`
   - 构建 R6 所属测试目标，确认新增测试没有编译错误。

5. `./bin/test_ai_gateway_real_provider`
   - RED 阶段先运行测试，确认不完整录制响应被错误当作成功响应，且假响应中的敏感文本会进入成功结果；这是 R6 要修的真实缺口。

6. `cmake --build build --target test_ai_gateway_real_provider`
   - 实现形状校验后重新构建目标，确认生产代码和测试代码可编译。

7. `./bin/test_ai_gateway_real_provider`
   - GREEN 阶段验证 R6 新增的录制响应成功/失败用例通过，并确认旧的真实 Provider 回归没有被破坏。

8. `cmake --build build`
   - 按仓库 C++ 源码变更验证规则做完整构建，确认 AI gateway module 和相关测试目标都能编译。

9. `ctest --test-dir build -R ai_gateway_real_provider --output-on-failure`
   - 验证 CTest 注册的真实 Provider 精确测试入口通过。

10. `git diff --check`
   - 检查本次源码、测试、文档改动没有尾随空白或补丁格式问题。

11. `ctest --test-dir build -L unit --output-on-failure`
    - 做全量 unit 标签回归。当前沙箱下 `http_attempt_outcome` 和 `ai_gateway_load_balance` 失败，失败日志为 `socket(2, 1, 0) errno=1 errstr=Operation not permitted`，属于本地 loopback socket 创建受限；`ai_gateway_real_provider` 在该轮回归中通过。

## 结论

R6 使用本地录制响应完成基础闭环：合法 OpenAI-compatible 非流式响应会被 adapter 接受并归一为网关响应；缺少基础字段的不完整响应会进入脱敏的 `RESPONSE_PARSE_FAIL` 路径，不回显 provider 原始 body、API Key 或用户消息。真实 Provider 访问仍默认关闭，未引入外网依赖。
