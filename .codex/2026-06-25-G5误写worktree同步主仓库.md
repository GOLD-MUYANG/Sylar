# G5 误写 worktree 后同步主仓库

## 背景

G5 状态接口最初实现到了 `.worktrees/ai-gateway-g3`，但用户当前需要的是主仓库
`/home/muyang/workspace/sylar`。主仓库的 `sylar/http/http_load_balance_client.h`
仍是旧版本，所以 IDE 在 `ai_gateway_status_servlet.cc` 中提示
`getStatusSnapshots()` 不存在。

## 排查命令

1. `git status --short`
   - 目的：确认主仓库当前是否有未提交改动，避免同步时覆盖用户已有修改。
   - 结果：同步前主仓库干净。

2. `rg -n "getStatusSnapshots|HttpEndpointStatusSnapshot|AiGatewayStatusServlet|internal/status" ...`
   - 目的：确认主仓库是否已经存在 G5 状态接口相关代码。
   - 结果：主仓库只有 G4 文档和注释提到 `/internal/status`，没有状态 Servlet 和快照接口。

3. `sed -n ... CMakeLists.txt modules/ai_gateway/ai_gateway_module.cc sylar/http/http_load_balance_client.* 文档/AI模型调用网关方案.md`
   - 目的：读取主仓库现状，确认主仓库已有 G4 的 limiter、breaker、health_check、max_total_attempts，不能用 worktree 老 diff 覆盖。
   - 结果：主仓库已有 G4，需要只做 G5 增量合并。

4. `cmake -S . -B build && cmake --build build --target test_ai_gateway_status ai_gateway_module && ./bin/test_ai_gateway_status`
   - 目的：验证主仓库新增 G5 状态接口能编译，且 `ai_gateway_status_servlet.cc` 能看到主仓库里的 `getStatusSnapshots()`。
   - 结果：构建和测试通过。

5. `cmake --build build --target test_ai_gateway_load_balance && ctest --test-dir build -L unit --output-on-failure`
   - 目的：验证 G5 增量没有破坏已有 G4/AI gateway 单元回归。
   - 结果：12 个 unit 测试全部通过。该命令需要本地 loopback socket，因此提升权限运行。

6. `./scripts/demo_ai_gateway.sh`
   - 目的：验证主仓库 demo 能访问 `/internal/status`，并展示停止 `mock-a` 后故障转移到 `mock-b`。
   - 结果：demo 通过，状态接口显示 `mock-a.failure_count=1`、`mock-b.success_count=2`。

## 结论

G5 已从 worktree 同步到主仓库，并按主仓库已有 G4 代码做了增量合并；主仓库现在包含
`getStatusSnapshots()`、`AiGatewayStatusServlet`、`/internal/status` 路由、G5 测试、
请求样例、demo 输出和方案文档状态更新。
