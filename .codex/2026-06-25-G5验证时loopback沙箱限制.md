# G5 验证时 loopback 沙箱限制

## 背景

实现 G5 后需要验证 AI Gateway 单元测试和一键演示脚本。其中
`test_ai_gateway_load_balance` 和 `scripts/demo_ai_gateway.sh` 都会创建本地
loopback HTTP 服务。

## 排查命令

1. `ctest --test-dir build -L unit --output-on-failure`
   - 目的：先按常规单元测试入口验证 G5 是否影响已有回归。
   - 结果：只有 `ai_gateway_load_balance` 失败，日志显示 `socket(2, 1, 0) errno=1 errstr=Operation not permitted`。
   - 判断：失败发生在创建本地 socket 阶段，不是 HTTP 负载均衡或 G5 状态接口断言失败。

2. `ctest --test-dir build -L unit --output-on-failure`（提升权限重跑）
   - 目的：排除沙箱对 loopback socket 的限制，验证真实测试结果。
   - 结果：12 个 unit 测试全部通过。

3. `./scripts/demo_ai_gateway.sh`（提升权限运行）
   - 目的：验证 G5 的 `/internal/status` 与 G3 轮询/故障转移演示能端到端跑通。
   - 结果：脚本启动两个 Mock Provider 和网关成功；停止 `mock-a` 后第三次请求故障转移到 `mock-b`，状态接口显示 `mock-a.failure_count=1`、`mock-b.success_count=2`。

## 结论

未提升权限时的失败来自当前执行环境禁止本地 socket，而不是本次 G5 代码问题。
