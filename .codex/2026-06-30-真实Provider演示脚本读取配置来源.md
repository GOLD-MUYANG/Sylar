# 2026-06-30 真实Provider演示脚本读取配置来源

## 问题

`scripts/demo_ai_gateway_real_provider.sh` 运行时通过 heredoc 生成
`$tmp_conf/ai_gateway.yml`，导致 AI Gateway 模块实际读取的是临时 YAML，
不是维护中的 `examples/ai_gateway_conf/ai_gateway.yml`。

## 排查命令和原因

- `rg -n "getConfigPath|ai_gateway.yml|YAML::LoadFile" sylar modules scripts examples`
  - 确认 C++ 模块读取配置的路径来自 `EnvMgr::getConfigPath() + "/ai_gateway.yml"`。
- `sed -n '1,220p' sylar/env.cc`
  - 确认 `-c` 参数决定配置目录；相对路径按启动目录解释。
- `sed -n '1,260p' modules/ai_gateway/ai_gateway_module.cc`
  - 确认 mock gateway 和 real provider 都读取同一个 `ai_gateway.yml`。
- `sed -n '1,180p' scripts/demo_ai_gateway_real_provider.sh`
  - 确认脚本用临时目录和 heredoc 绕开了仓库里的 YAML。
- `rg -n "SYLAR_AI_GATEWAY_REAL_...|cat >\"\$tmp_conf/ai_gateway.yml\"" scripts/demo_ai_gateway_real_provider.sh`
  - 确认脚本内存在真实 provider 配置默认值和临时 YAML 生成逻辑。

## 修复

把 `scripts/demo_ai_gateway_real_provider.sh` 收紧为启动器：

- 不再生成 `/tmp/.../ai_gateway.yml`。
- 不再写死真实 provider 的 base_url、chat_path、upstream_model、api_key_env、tls_server_name。
- 直接使用 `./bin/sylar -s -c examples/ai_gateway_conf`，让模块读取
  `examples/ai_gateway_conf/ai_gateway.yml`。

后续 DeepSeek、Ark、模型名、`api_key_env`、provider enabled 状态都以 YAML 为准。
