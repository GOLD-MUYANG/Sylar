# 真实 Provider 页面请求 504 排查

## 现象

- 页面 `/demo` 点击“发送请求”后返回 HTTP 504。
- trace 显示：
  - `request_received`
  - `model_parsed`
  - `candidates_selected`
  - `attempt_result`
  - `response_returned`
- 失败尝试为 `volcengine-ark`，phase 为 `response_timeout_after_send`，错误分类为
  `PROVIDER_RESULT_UNKNOWN`。

## 排查命令和原因

1. `curl -sS --max-time 3 http://127.0.0.1:18080/internal/status`
   - 原因：确认页面状态是否与网关内部状态一致。
   - 结果：普通沙箱网络无法连接宿主服务，随后使用提升权限访问本机端口。

2. `curl -sS --max-time 3 http://127.0.0.1:18080/internal/status`
   - 原因：读取真实运行中的网关 provider 状态。
   - 结果：`volcengine-ark` failure_count 为 1，last_failure_reason 为
     `PROVIDER_RESULT_UNKNOWN`；两个 mock provider success/failure 均为 0。

3. `ps -ef | rg "bin/sylar|mock_model_provider|demo_ai_gateway_real_provider"`
   - 原因：确认脚本、网关和两个 mock provider 都在运行。
   - 结果：脚本、`mock-provider-a`、`mock-provider-b`、`./bin/sylar -s -c /tmp/tmp.f8iIXt5E5d`
     均在运行。

4. `nl -ba /tmp/tmp.f8iIXt5E5d/ai_gateway.yml | sed -n '1,120p'`
   - 原因：确认脚本生成的临时配置。
   - 结果：`request_deadline_ms: 30000`，真实 provider 排在两个 mock provider 之前。

5. `curl -sS --max-time 3 -X POST http://127.0.0.1:19101/v1/chat/completions ...`
   - 原因：确认本地 mock provider 是否可用。
   - 结果：`mock-provider-a` 立即返回 mock completion，说明 mock 本身正常。

6. `tr '\0' '\n' </proc/449330/environ | sed -n 's/^ARK_API_KEY=.*/ARK_API_KEY=SET/p'`
   - 原因：只确认网关进程是否拿到了 Key，不打印 Key。
   - 结果：`ARK_API_KEY=SET`。

7. `curl -sS --connect-timeout 5 --max-time 10 -X POST https://ark.cn-beijing.volces.com/api/v3/chat/completions ...`
   - 原因：无 Key 验证 Ark 域名、TLS、路径是否可达。
   - 结果：HTTP 401，耗时约 0.2 秒，说明端点可达。

8. 使用 `/proc/449330/environ` 中的 `ARK_API_KEY` 生成临时 header 文件后直连 Ark。
   - 原因：不打印 Key 的前提下验证真实 provider 请求耗时。
   - 结果：HTTP 200，耗时约 33.0 秒。

## 当前结论

- 页面未成功不是因为 `/demo` 没发请求，也不是脚本没拿到 Key。
- 真实 Ark 请求本身耗时约 33 秒，超过脚本生成配置中的 `request_deadline_ms: 30000`。
- 网关在已发送请求后等待响应超时，会标记为 `PROVIDER_RESULT_UNKNOWN`，这是为了避免重复提交语义风险，因此不会继续尝试后面的 mock provider。

## 建议后续

- 将 `scripts/demo_ai_gateway_real_provider.sh` 的真实演示 deadline 改为可配置，并把默认值调高到 60000ms 或更高。
- 同步更新 `文档/真实Provider演示网关任务书.md`，说明 live provider 慢响应时需要更大的 deadline，且 `response_timeout_after_send` 不会自动切换 mock。

## 补充验证：短请求可以成功

用户要求先判断“是否确实可以把请求发过去并拿到结果”。补充做了两层验证：

1. 直连 Ark：
   - 命令思路：使用当前宿主环境里的 `ARK_API_KEY` 生成临时 header 文件，不打印 Key；
     对 `https://ark.cn-beijing.volces.com/api/v3/chat/completions` 发送
     `{"model":"doubao-seed-2-0-lite-260428","messages":[{"role":"user","content":"只回答 OK"}],"stream":false}`。
   - 原因：绕过网关，确认真实上游是否能收到请求并返回。
   - 结果：HTTP 200，耗时约 1.55 秒，响应内容包含 assistant `OK`。

2. 通过网关入口：
   - 命令思路：启动 `./scripts/demo_ai_gateway_real_provider.sh`，再请求
     `http://127.0.0.1:18080/v1/chat/completions`，请求体使用
     `model=general-chat` 和 `content=只回答 OK`，并带 `X-Ai-Gateway-Demo-Trace: 1`。
   - 原因：确认网关能把请求转发给真实 Provider，并把真实结果转成 OpenAI-compatible 响应。
   - 结果：HTTP 200，耗时约 2.15 秒；trace 显示 `provider=volcengine-ark`、
     `attempt_phase=response_received`、`http_status=200`、`try_next_candidate=false`；
     响应 body 中 `choices[0].message.content` 为 `OK`。

3. `/internal/status`：
   - 原因：确认网关内部 provider 计数。
   - 结果：`volcengine-ark` 的 `success_count=1`，`failure_count=0`。

补充结论：真实 Provider 链路是通的，短请求可以正常发到 Ark 并通过网关拿到结果；之前页面失败的主要原因是那次请求耗时超过网关 30 秒 deadline。
