# AI 网关模块配置加载时机排查

## 结论

动态模块在 `Application::init()` 中的装载发生在首次 `Config::LoadFromConfDir()` 之后。网关模块里的 `Config::Lookup` 因此会在 YAML 已处理完才注册，`ai_gateway.yml` 的值不会写入该模块配置项，只能得到默认值。

这会使网关插件即使被成功加载，也无法可靠读取 `enabled`、`server_name`、`provider_url` 和超时配置。修复应保持模块配置归模块所有：首次加载基础配置以取得 `module.path`，装载模块后再次加载同一配置目录，使新注册的配置项接收 YAML 值。

## 使用的命令与原因

1. `sed -n '1,320p' sylar/application.cc`
   - 检查配置加载、`ModuleMgr::init()` 和模块回调的实际先后顺序。
2. `sed -n '1,300p' sylar/module.cc`
   - 确认模块由 `dlopen` 后才进入 `onLoad()`，并非静态链接期注册。
3. `rg -n -i 'module\.path|module:' bin/conf sylar tests README.md 文档`
   - 确认 `module.path` 是已有配置项，而网关配置尚无框架内静态注册位置。
4. `sed -n '1,180p' sylar/env.cc`
   - 确认模块相对路径基于可执行文件目录解析，便于后续演示配置写出正确的模块目录。

## 验证计划

- 为模块配置重载补专用测试：首次配置加载后注册新 `ConfigVar`，第二次加载相同目录后断言它读到 YAML 值。
- 启动网关时使用 `ai_gateway.yml`，断言插件可通过 `enabled=true`、目标服务名和 provider URL 成功注册路由。

## 端到端启动异常追加（待定位）

使用 `./bin/sylar -s -c examples/ai_gateway_conf` 启动时，进程在模块装载日志后退出，未监听 `127.0.0.1:18080`。已确认 `sylar/library.cc` 现有实现会在每个模块 `dlopen` 后调用一次 `Config::LoadFromConfDir(..., true)`；本轮为“所有模块加载完成后再统一应用模块配置”在 `Application::init()` 增加了第二处重载，两者可能重复且顺序不一致。

追加排查命令：

1. `ss -ltn | rg ':(18080|19001)\\b'`
   - 检查网关与 Mock Provider 是否实际进入监听。
2. `sed -n '140,210p' sylar/application.cc`
   - 核对 Application 的首次配置、模块装载和统一重载顺序。
3. `sed -n '1,130p' sylar/library.cc`
   - 核对动态库装载路径是否已有隐式配置重载。
4. `find bin/module -maxdepth 1 -type f -name '*.so' -printf '%f\\n' | sort`
   - 确认当前扫描到的模块集合，排除遗漏或错误模块。

5. `gdb -batch -ex run -ex bt --args ./bin/sylar -s -c examples/ai_gateway_conf`
   - 捕获无日志退出的真实信号与调用栈。

调试器确认进程在退出期收到 `SIGSEGV`：`Config` 的全局 `map<string, shared_ptr<ConfigVarBase>>` 析构时释放了由 `libai_gateway_module.so` 注册的 `ConfigVar<AiGatewayConfig>`；但模块先被 `ModuleCloser` 的 `dlclose` 卸载，导致该对象的代码/类型信息不再有效。

修复决策：卸载型业务插件不向核心 `Config` 全局表注册模块私有类型。AI 网关模块改为在 `onServerReady()` 直接从当前配置目录读取 `ai_gateway.yml`；`Application` 中新增的统一强制重载不再需要，应撤回以避免与 `library.cc` 已有的模块装载后重载重复。通用 `ConfigVar<bool>` 对 YAML `true/false` 的转换缺陷和回归测试保留，它是独立发现的框架配置修复。

## 示例服务配置兼容性追加

重新启动后的日志显示 `http_servers` 配置转换失败，随后网关模块报告 `ai gateway server not found name=ai-gateway`。根因是 `HttpServerConf` 的 YAML 转换对 `accept_worker`、`process_worker` 调用了无默认值的 `as<string>()`，两字段实际是当前实现中的必填项。示例 `server.yml` 已补齐 `accept_worker: accept` 与 `process_worker: io`，对应 `worker.yml` 中的两个 Worker。

追加排查命令：

1. `./bin/sylar -s -c examples/ai_gateway_conf`
   - 获取实际配置转换错误和模块 `onServerReady()` 失败原因。
2. `sed -n '1,320p' sylar/application.cc`
   - 确认 `HttpServerConf` 的字段读取方式及服务创建顺序。
