# AI 模型调用网关方案草案

> 状态：待审阅。本文件只定义项目目标、边界和可独立验证的实现切片，尚未开始修改框架或新增业务代码。
>
> 目标：在现有 Sylar HTTP 客户端能力之上，做一个**本地可运行、可注入故障的 AI 模型调用网关**。它不试图实现大模型推理；它要展示的是网关如何稳定地调用多个模型提供者。

## 1. 项目要解决什么问题

用户向网关发送一次“生成文本”请求。网关背后配置多个模型提供者实例：正常时分摊请求，某个实例超时、报错或被限流时自动切换；持续失败的实例被熔断，恢复后再被谨慎放回流量中。

```text
调用方（curl / 演示页面）
        |
        v
AI Gateway（Sylar Application + ai_gateway 插件）
        |
        | HttpLoadBalanceClient
        +----> Mock Model A :9001
        +----> Mock Model B :9002
        +----> Mock Model C :9003（可选）
```

这里的 `Mock Model` 是本地的“模型提供者模拟服务”，按配置返回固定文本、延迟或错误。第一版不用真实 OpenAI、DeepSeek 等 API，也不需要 API Key、外网或 GPU。

这样做的原因是：项目要诚实地展示**服务调用治理**，而不是把模型厂商的推理能力误写成自己的能力。

## 2. 为什么它能用到现有能力

| 现有模块 | 在网关中的位置 | 具体作用 |
| --- | --- | --- |
| `HttpConnectionPool` | 到每个模型提供者的连接 | 复用连接，避免每次请求重新建连 |
| `HttpClient` | 单个模型实例请求 | 超时、重试和错误归一化 |
| `HttpLoadBalanceClient` | 多个模型实例之前 | 选择实例，并在一次调用内故障转移 |
| `HttpConcurrencyLimiter` | 出站调用准入边界 | 在建立下游请求前限制全部/某个模型服务/某个实例的并发和 QPS，避免网关和供应商被突发流量拖垮 |
| `HttpCircuitBreaker` | 每个模型实例 | 连续失败时暂时隔离实例；冷却后半开探测 |
| `HttpServer` / `ServletDispatch` | 网关入口 | 接收调用方请求，返回统一响应 |
| Worker / IOManager / 协程 | 运行基础 | 承载连接、定时健康检查与并发请求 |

注意：当前 `HttpConcurrencyLimiter` 是**出站客户端限流**，必须作为第一版网关的核心保护；它不是“阻止大量用户进入网关”的入口限流。模型供应商自己的 429/排队只能保护供应商，而在收到该结果前，网关已经消耗了自身的协程、连接、内存和等待时间。

第一版的策略是**有界准入，不做无界排队**：有名额就立即转发；某个实例达到上限时由负载均衡器尝试其他实例；所有候选实例都无法准入时快速返回 `RATE_LIMITED`。这比无限等待更能保护延迟和吞吐。按用户身份的入口配额不是第一版范围，但以后如需加入，也必须作为独立能力，不能混进现有 limiter 的职责里。

## 3. 当前已经具备什么，还缺什么

### 已具备

- 单实例 HTTP 请求、连接池、超时、重试和错误归一化。
- 多实例选择：轮询、随机、加权轮询、最少连接、健康检查。
- 同一次请求内的故障转移。
- 出站并发/QPS 限制，以及 endpoint 级熔断器。
- HTTP Server 和 Servlet 路由分发能力。
- 动态模块的基础类型与加载机制。

### 必须补齐的前置能力

当前 `Application` 创建 `HttpServer` 后直接注册了 `/sylarx/xx` 示例路由；虽然 `Module` 声明了 `onLoad()`、`onServerReady()`、`onServerUp()`，模块还不能可靠地在“服务器已创建、尚未开始接收请求”这个阶段取得 Server 并注册业务路由。

因此 AI 网关之前需要先完成两项小而明确的框架接入：

1. **模块生命周期可靠化**：落实 `onLoad()` / `onUnload()` 的调用与失败处理。这对应现有问题拆分方案中的 F04。
2. **业务路由注册扩展点**：`Application` 在启动 HTTP Server 后、`start()` 前，调用模块的 `onServerReady()`；同时提供只读的 HTTP Server 获取接口或等价的受控注册器，让模块能向 `ServletDispatch` 添加路由。

第二项只解决“模块可以注册路由”，不引入热插拔、不让模块任意接管 Application，也不把业务代码塞回 `application.cc`。

### 新增但保持独立的业务组件

| 组件 | 责任 | 不负责什么 |
| --- | --- | --- |
| `ai_gateway_module` | 装配网关路由、读取网关配置 | 不实现负载均衡或熔断状态机 |
| `AiGatewayServlet` | 校验请求、调用客户端、组装响应 | 不管理 endpoint 选择细节 |
| 协议/响应类型 | 定义最小请求、响应和错误 JSON | 不保存业务数据或用户系统 |
| `mock_model_provider` | 模拟一个模型服务的正常、慢、错误行为 | 不伪装成真实大模型 |
| 网关观测组件 | 统计请求、选中的实例、失败转移、拒绝原因 | 不在第一版接 Prometheus、ELK 等外部系统 |

## 4. 最小对外接口

建议采用一个接近 OpenAI 风格、但刻意缩小的接口：

```http
POST /v1/chat/completions
Content-Type: application/json

{
  "model": "demo-chat",
  "prompt": "用一句话解释熔断"
}
```

正常响应：

```json
{
  "request_id": "...",
  "model": "demo-chat",
  "provider": "mock-a",
  "content": "mock-a: 熔断会暂时隔离持续失败的下游服务。"
}
```

当所有提供者均不可用时，返回统一的 5xx 错误和可读错误码，例如 `UPSTREAM_UNAVAILABLE`；不会把内部地址、堆栈或凭据暴露给调用方。

### JSON 的选择

仓库当前未见已接入的专用 JSON 依赖。实现阶段需要单独确认下列方案之一：

- 推荐：引入一个明确、隔离的 JSON 依赖（例如 `jsoncpp` 或 header-only JSON 库），只放在网关协议层。
- 保守：第一阶段使用极简受控 JSON 字段解析，随后再替换为正式库。

不建议在 `HttpRequest`、`HttpResponse` 或负载均衡代码里散落字符串拼接解析。

## 5. 运行时行为：如何演示已有能力

| 演示场景 | 提供者状态 | 期望行为 | 证明的能力 |
| --- | --- | --- | --- |
| 正常分流 | A、B 正常 | 请求在 A、B 间按策略分配 | 负载均衡 |
| 慢实例 | A 延迟超过超时 | 当前请求改尝试 B | 超时、重试/故障转移 |
| 连续失败 | A 连续返回 5xx | A 被熔断，后续请求绕过 A | 熔断 |
| 恢复探测 | A 恢复正常 | 冷却后以半开请求验证，再恢复流量 | 半开状态 |
| 下游配额保护 | B 达到并发/QPS 上限 | 网关不再向 B 建立请求，优先尝试其他实例 | 出站限流与 token 归还 |
| 网关突发保护 | 短时间请求超过全局并发上限 | 不建立无界等待队列，快速拒绝或切换 | 协程、连接、内存保护 |
| 全部不可用 | A、B 都失败 | 返回结构化失败，不崩溃、不无限重试 | 错误边界 |

为了让这些场景可重复，模拟服务应有**仅本地演示使用**的控制方式，例如启动参数或 `/__control` 路由，用于切换 `normal`、`slow`、`error` 三种状态。该控制接口不能作为正式网关 API 对外宣传。

## 6. 推荐实施顺序

每一步都是可独立验证、可单独审阅的小修复/小功能；前一步不通过，不进入后一步。

### G0：补齐插件装配入口（前置）

- 完成 F04 的模块 load/unload 语义。
- 新增受控的 HTTP Server/路由注册入口，并在 `onServerReady()` 调用它。
- 验证：一个最小测试模块在启动时注册 `/module-ping`，请求得到预期响应；模块加载、卸载次数可断言。

### G1：实现可控的模拟模型提供者

- 新增独立可执行程序，而非塞进 `Application`。
- 一个进程对应一个端口和 provider 名称；至少支持正常响应、固定延迟、固定 5xx。
- 验证：分别启动 A、B，使用 curl 能区分返回的 provider 名和状态。

### G2：实现网关最小直连链路

- 新增 `ai_gateway_module`，注册 `POST /v1/chat/completions`。
- 先只配置一个 provider，通过 `HttpClient` 转发并返回统一 JSON。
- 验证：网关请求成功；非法请求、上游超时、上游 5xx 都有稳定且可读的结果。

### G3：接入多实例负载均衡和故障转移

- 将 G2 的单实例客户端替换为 `HttpLoadBalanceClient`，配置 A、B 两个 endpoint。
- 先验证正常轮询，再验证 A 失败时同一业务请求落到 B。
- 验证：运行日志和响应中的 provider 能清楚证明请求路径。

### G4：接入 limiter、熔断与健康检查

- 根据网关可承受的并发量和模型供应商公开的并发/RPM 限额，配置全局、服务级和 endpoint 级出站上限；配置值不是固定常数，必须通过压测和供应商配额确定。
- 准入失败时不排入无界队列：当前候选实例满额则继续尝试其他实例；全部候选都满额时快速返回 `RATE_LIMITED`，释放当前请求占用的资源。
- 通过配置开启 endpoint 熔断器和健康检查；它们用于跳过已失败的实例，避免无谓等待，属于核心链路。
- 严格设置单实例请求重试次数；不在网关 Servlet 外再套无限重试循环。一次业务请求只能尝试有限个 endpoint，防止失败重试形成请求风暴。
- 配置健康检查，并针对三种故障状态建立回归测试。
- 验证：按第 5 节故障矩阵逐项演示；压力超过上限时，网关的 in-flight 数保持有界，且请求不会无限等待。

### G5：补最小可观测性与演示材料

- 增加只读 `/internal/status` 或 `/internal/metrics`，显示每个 provider 的健康状态、熔断状态、成功/失败/限流计数。
- 增加一键演示脚本、配置样例、请求样例和故障切换说明。
- 可选：增加极简静态 HTML 页面；它只是演示入口，不应成为项目的重心。

### G6：可选的真实模型适配器（最后再做）

- 仅在 G0–G5 的本地 Mock 闭环、测试和故障演示稳定后再开始；它不能阻塞离线演示，也不能让 CI 依赖外网或真实 API Key。
- 先接入一个兼容 OpenAI Chat Completions 形状的远端提供者；第二个供应商只有在协议不兼容时才增加专用适配器。
- 实现边界、前置安全要求、错误策略和验证方式见第 8 节。

## 7. 配置草图

配置结构只表达意图，字段名在实现时再与现有 `Config` 序列化方式对齐：

```yaml
ai_gateway:
  route: /v1/chat/completions
  request_timeout_ms: 800
  load_balance: WEIGHTED_ROUND_ROBIN
  providers:
    - name: mock-a
      url: http://127.0.0.1:9001
      weight: 2
    - name: mock-b
      url: http://127.0.0.1:9002
      weight: 1
  # 示例值，实际值由网关压测结果和供应商配额决定。
  limiter:
    max_global_concurrency: 32
    max_global_qps: 100
    max_endpoint_concurrency: 8
    max_endpoint_qps: 20
  circuit_breaker:
    enabled: true
    failure_threshold: 3
```

配置职责保持分开：endpoint/权重属于负载均衡；超时与单实例重试次数属于请求选项；并发/QPS 准入属于 limiter；失败阈值和冷却时间属于 breaker。真实模型供应商若还按 token/分钟计费或限额，现有 QPS limiter 只能做基础保护，token 配额需要后续独立设计。

## 8. 真实 Provider 的可选接入（G6）

### 8.1 前置条件

真实模型 API 通常使用 HTTPS 和 API Key。因此在开始 G6 前，必须满足：

1. 完成问题修复方案中的 F06：客户端启用可信 CA、主机名校验和 SNI；不能为了“先连通”而关闭证书校验。
2. G0–G5 已在本地 Mock Provider 上稳定运行；真实 API 只增加一个可选 endpoint，不能替代本地测试夹具。
3. 网关协议层已使用正式 JSON 库；不能用字符串拼接承载外部不可信 JSON。
4. API Key 只能通过环境变量名或本地忽略配置引用；不得出现在仓库、日志、状态接口、错误响应或测试快照中。

现有 `HttpClient` 已支持请求头、请求体、超时和有限重试，因此适配器可以直接复用它；不需要为了调用某一家供应商而引入其 C++ SDK。

### 8.2 适配器边界

网关对外只维护一份内部协议，例如 `GatewayChatRequest` 与 `GatewayChatResponse`。新增一个 `AiProviderAdapter` 抽象，让不同 provider 各自负责：

- 把统一的网关请求转换为供应商的 URL、HTTP 头和 JSON 请求体；
- 从供应商响应中解析文本、模型名、usage 和错误信息；
- 把供应商 HTTP 错误分类为“可重试”“可故障转移”“用户请求错误”“认证/配置错误”等网关语义。

`mock_model_provider` 也应实现同一套适配器契约，而不是绕过网关协议层。这样 G6 的重构能先由本地测试覆盖，不会把真实供应商变成唯一验证手段。

推荐首先提供 `OpenAICompatibleProvider`：它通过配置的 base URL、chat path、模型名和 API Key 环境变量工作。只有当某个供应商的认证或响应结构确实不兼容时，才新增单独的 provider adapter。

### 8.3 模型路由与负载均衡边界

不能把任意模型或任意供应商直接混进同一个负载均衡池。网关应先定义“逻辑模型”，例如 `general-chat`，再为它配置一组**能力、上下文长度、输出格式和价格策略均可接受**的 provider/model 组合：

```text
调用方请求 general-chat
        ↓
模型路由表
        ↓
general-chat 的兼容 provider 池
        ↓
HttpLoadBalanceClient 选择一个 endpoint
```

只有同一逻辑模型池中的实例才允许自动负载均衡或故障转移。更昂贵、更强或语义不同的模型只能作为显式降级策略，不能被随机轮询命中。

### 8.4 真实供应商的错误与重试策略

接入真实 provider 前，需要在适配器层补一层错误分类，不能把所有非 2xx 都当成“后端故障”：

| 情况 | 网关动作 | 是否记入 breaker |
| --- | --- | --- |
| 请求格式错误、模型不存在（通常为 4xx） | 直接返回调用方，不重试 | 否 |
| API Key 缺失、失效或权限不足（401/403） | 返回安全的配置/认证错误，并告警 | 否 |
| 供应商限额（429） | 不无限重试；可尝试同池其他实例，全部不可用则快速失败 | 否，需单独记录 quota 状态 |
| 网络错误、连接超时、供应商 5xx | 在有限尝试预算内重试或故障转移 | 是 |

这一步是对现有调用链的必要优化：目前客户端能归一化 HTTP 状态和执行重试，但真实 provider 需要更细的业务错误语义。重试预算必须同时考虑“单实例重试次数”和“同请求可尝试 endpoint 数”，不能在 Servlet 外再包一层无限循环。

### 8.5 凭据、日志与观测

建议配置形状如下；`api_key_env` 保存的是环境变量名，而不是 Key 本身：

```yaml
real_providers:
  - name: provider-a-chat
    enabled: false
    type: openai_compatible
    logical_model: general-chat
    base_url: https://api.provider.example
    chat_path: /v1/chat/completions
    upstream_model: provider-a-model
    api_key_env: PROVIDER_A_API_KEY
    timeout_ms: 15000
    max_retry: 1
```

- 默认 `enabled: false`；没有设置对应环境变量时，启动阶段给出明确但不泄密的错误，不尝试对外连接。
- 日志仅记录 request id、provider 名、耗时、HTTP 类别、尝试次数和限流/熔断状态；不记录 `Authorization`、API Key、完整 prompt 或完整模型响应。
- `/internal/status` 只展示脱敏后的 provider 名和运行状态，不能回显 base URL 中的敏感查询参数或环境变量内容。
- 真实模型常按 token/分钟限额和计费。现有并发/QPS limiter 仍会启用，但只能作为第一层保护；token 估算、日预算和费用治理是后续独立特性，不在 G6 内假装完成。

### 8.6 G6 的独立实施切片与验证

| 切片 | 修改范围 | 独立验证 |
| --- | --- | --- |
| G6.0 | 完成 F06 的 TLS 安全默认值 | 受信 CA 成功；错误主机名和未信任证书失败 |
| G6.1 | 引入统一请求/响应类型和 `AiProviderAdapter`，让 Mock 也走适配器 | Mock 的原有正常、超时、错误用例全部保持通过 |
| G6.2 | 实现一个非流式 `OpenAICompatibleProvider` | 使用本地录制响应或假 HTTPS 服务做适配器契约测试 |
| G6.3 | 增加供应商错误分类、有限重试预算和逻辑模型路由 | 断言 401/403/429 不错误打开 breaker，5xx/超时可触发故障转移 |
| G6.4 | 手工真实 smoke test | 显式设置 API Key 环境变量后请求一次；不进入默认测试或 CI |

流式输出（SSE）不属于 G6 的必做范围。它会让一次模型请求长期占用连接和 limiter 名额，需要单独确认响应流读取、客户端断连、permit 释放和背压语义后，再作为新的设计项处理。

## 9. 明确不做的内容

- 不训练、不部署、不宣称实现了大语言模型。
- 第一版不做账号、计费、数据库、会话历史、向量数据库、RAG、流式 token 输出。
- 第一版不接 Kubernetes、服务注册中心、Redis、Prometheus 或真实云厂商。
- 不为了演示而修改 `HttpLoadBalanceClient`、limiter、breaker 的职责边界。
- 不实现无界请求队列，也不增加无限重试；满额或所有候选不可用时必须快速结束请求。
- 不把所有组件放进一个巨大的 `application.cc` 或单个“网关万能类”。

这些可以在 README 的“已知限制与后续方向”中诚实说明；它们不是第一版演示闭环的阻塞项。

## 10. 完成标准

完成 G0–G5 后，应能在不依赖外网、API Key、GPU 的条件下演示：

1. 启动两个本地模拟提供者和一个 Sylar 网关。
2. 向网关发起同一种请求，观察请求被分给不同 provider。
3. 让一个 provider 变慢或失败，观察当前请求故障转移。
4. 连续制造失败，观察熔断打开、冷却、半开和恢复。
5. 将请求压力提高到某个实例或网关全局上限，观察不再建立无界等待，并优先切换其他可用 provider。
6. 通过状态接口和日志解释每次路由、限流拒绝、熔断跳过或故障切换的原因。
7. 每个核心行为都有对应的专用测试，不需要真实模型服务。

## 11. 审阅时需要确认的决策

1. 是否接受“本地 Mock Model + 可选真实适配器”的边界？推荐接受，它让演示稳定、可复现，也不夸大项目能力。
2. 是否先做 G0，再按 G1 → G5 推进？推荐接受；G0 是业务插件不再写死在 `Application` 的必要地基。
3. JSON 依赖倾向引入正式库，还是先用受控极简实现？推荐正式库，但在真正改 CMake 前再单独审阅依赖选择。
4. 是否确认 limiter 默认开启，并采用“有界准入、候选切换、全部满额后快速失败”的策略？推荐确认；它保护的是网关与下游，而不是为用户请求增加无意义的排队。
5. 是否确认 G6 只在本地 Mock 闭环完成后开始，并以一个非流式 OpenAI 兼容 provider 为首个真实适配器？推荐确认。
