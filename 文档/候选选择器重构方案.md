# 候选选择器重构方案

> 状态：已实现。HTTP 与真实 Provider 已统一复用 `CandidateSelector<T>`，离线 selector、
> real-provider executor/runtime 回归已接入构建；loopback HTTP 测试仍受运行环境 socket 权限约束。
>
> 目标：把负载均衡选择算法从 `HttpLoadBalanceClient` 中抽取为通用
> `CandidateSelector<T>`，同时用于 `HttpEndpoint::ptr` 和真实 Provider 的
> `ProviderCandidate`，并让两条链路统一支持四种现有策略。

## 1. 背景与问题

当前项目存在两条候选选择链路。

通用 HTTP / Mock Provider 路径使用：

```text
AiGatewayProviderConfig
        ↓
CreateLoadBalanceClient()
        ↓
HttpEndpoint
        ↓
HttpLoadBalanceClient
        ↓
ROUND_ROBIN / RANDOM / WEIGHTED_ROUND_ROBIN / LEAST_CONNECTION
```

真实 Provider 路径使用：

```text
ProviderCandidate
        ↓
RealProviderRuntime::OrderCandidates()
        ↓
LogicalModelRouter
        ↓
ProviderAttemptExecutor
        ↓
AiProviderAdapter
        ↓
HttpConnection::DoPost()
```

第一条链路已经实现四种负载均衡策略，但算法与 `HttpEndpoint`、连接活跃数更新和
HTTP 请求生命周期写在 `HttpLoadBalanceClient` 内部。第二条链路需要先选出完整的
`ProviderCandidate`，才能根据候选自己的 adapter、path、upstream model、API Key 引用和
TLS 配置构造请求，因此不能直接把真实 Provider 塞进只接受统一 path、headers 和 body 的
`HttpLoadBalanceClient`。

目前真实 Provider runtime 只实现了 ROUND_ROBIN，而且实现方式是每个业务请求旋转候选列表
起点。`ProviderCandidate::weight` 虽然已经进入配置和类型，但没有参与选择；RANDOM、
WEIGHTED_ROUND_ROBIN 和 LEAST_CONNECTION 也没有接入真实 Provider 路径。

本次重构解决的是“复用选择算法”，不合并 HTTP endpoint 与真实 Provider 的请求执行语义。

## 2. 目标与非目标

### 2.1 目标

1. 新增与 HTTP、AI gateway 都无关的 `CandidateSelector<T>` 公共组件。
2. 提供 ROUND_ROBIN、RANDOM、WEIGHTED_ROUND_ROBIN、LEAST_CONNECTION 四种实现。
3. `HttpLoadBalanceClient` 使用公共 selector，保留原有连接池、健康状态、限流、熔断和统计职责。
4. `RealProviderRuntime` 和 `ProviderAttemptExecutor` 使用同一套 selector，四种策略均可配置。
5. 每次故障转移都重新选择，并通过 `tried_keys` 保证同一业务请求不重复选择同一候选。
6. selector 读取候选的最新可用性、权重和 active 数，不负责资源占用与释放。
7. 保持真实 Provider 的共享预算、错误分类和非幂等重试安全边界不变。
8. 保持现有策略配置名称和 `HttpLoadBalanceStrategy::XXX` 调用形式兼容。

### 2.2 非目标

1. 不把 `ProviderCandidate` 包装成 `HttpEndpoint`。
2. 不让通用 selector 负责 HTTP 请求、adapter、限流、熔断或错误响应生成。
3. 不实现动态增删 endpoint/provider；本轮候选列表仍来自启动配置。
4. 不改变 WEIGHTED_ROUND_ROBIN 当前“按权重连续返回”的语义，不引入 smooth weighted round robin。
5. 不实现模型能力自动发现，也不改变 `compatibility_key` 的人工声明机制。
6. 不改变 Provider 401/403、429、5xx、可能已提交请求的故障转移规则。
7. 不修改 `文档/手动理解项目.md` 和 `文档/A_代码相关问题集` 中用户正在编辑的内容。

## 3. 公共组件设计

### 3.1 文件边界

新增：

```text
sylar/load_balance/candidate_selector.h
sylar/load_balance/candidate_selector.cc
```

`candidate_selector.h` 负责：

- 通用策略枚举；
- 候选元数据访问器；
- `CandidateSelector<T>` 接口；
- 四种模板 selector；
- 模板 factory。

模板实现必须放在头文件中。文件内按照“公共类型、公共辅助函数、四种算法、factory”分块，并只在
状态推进、候选过滤和并发保护等关键位置添加简洁注释。

`candidate_selector.cc` 只负责不依赖模板的功能：

- `ParseLoadBalanceStrategy()`；
- `LoadBalanceStrategyToString()`。

### 3.2 策略枚举

```cpp
namespace sylar
{
namespace load_balance
{

enum class LoadBalanceStrategy
{
    ROUND_ROBIN = 0,
    RANDOM = 1,
    WEIGHTED_ROUND_ROBIN = 2,
    LEAST_CONNECTION = 3,
};

bool ParseLoadBalanceStrategy(const std::string &value,
                              LoadBalanceStrategy *strategy);
const char *LoadBalanceStrategyToString(LoadBalanceStrategy strategy);

} // namespace load_balance
} // namespace sylar
```

HTTP 层通过别名保持现有源码调用形式：

```cpp
namespace sylar
{
namespace http
{

using HttpLoadBalanceStrategy = sylar::load_balance::LoadBalanceStrategy;

} // namespace http
} // namespace sylar
```

因此已有代码仍然可以使用：

```cpp
HttpLoadBalanceStrategy::ROUND_ROBIN
HttpLoadBalanceStrategy::RANDOM
HttpLoadBalanceStrategy::WEIGHTED_ROUND_ROBIN
HttpLoadBalanceStrategy::LEAST_CONNECTION
```

### 3.3 候选元数据访问器

```cpp
template <class T>
struct CandidateAccessors
{
    std::function<std::string(const T &)> key;
    std::function<bool(const T &)> available;
    std::function<uint32_t(const T &)> weight;
    std::function<uint32_t(const T &)> active_requests;

    bool valid() const;
};
```

访问器负责告诉通用算法如何从不同对象读取选择所需信息。公共组件不包含
`HttpEndpoint` 或 `ProviderCandidate` 头文件。

factory 要求四个访问器全部存在。即使当前策略暂时不用某个字段，也必须提供完整访问器，避免
运行期间切换策略或重构后才触发空 `std::function` 调用。

### 3.4 Selector 接口

```cpp
template <class T>
class CandidateSelector
{
public:
    typedef std::shared_ptr<CandidateSelector<T>> ptr;

    virtual ~CandidateSelector() {}

    virtual bool select(const std::string &pool_key,
                        const std::vector<T> &candidates,
                        const std::vector<std::string> &tried_keys,
                        T *selected) = 0;
};
```

参数含义：

| 参数 | 含义 |
| --- | --- |
| `pool_key` | 隔离不同候选池的内部轮询状态；HTTP 使用固定 pool key，真实 Provider 使用 `logical_model` |
| `candidates` | 当前候选对象列表 |
| `tried_keys` | 本次业务请求已经尝试或已被治理组件拒绝的候选 key |
| `selected` | 接收本次选择出的候选 |

返回 `false` 的情况：

- `selected == nullptr`；
- accessors 不完整；
- 候选列表为空；
- 所有候选不可用；
- 所有候选都已进入 `tried_keys`；
- 剩余候选的 key 为空。

selector 不构造 HTTP 或网关错误。调用方根据自己的领域语义处理“无候选可选”。

## 4. 四种算法语义

### 4.1 ROUND_ROBIN

每个 `pool_key` 独立维护 `next_index`：

```text
pool-a: next_index = 1
pool-b: next_index = 0
```

每次从当前 pool 的 `next_index` 开始扫描一圈，跳过不可用、已尝试和 key 为空的候选。成功选中后，
把该 pool 的下一个位置推进到选中位置之后。

候选列表长度变化时，通过当前列表长度取模修正下标，避免越界。本轮不提供动态配置刷新，但该处理
可以让 selector 对测试或未来调用方更稳健。

### 4.2 RANDOM

先收集所有可用、未尝试且 key 非空的候选，再用 `std::mt19937` 和均匀分布选择一个。

不使用全局 `rand()`。生产 factory 使用 `std::random_device` 初始化随机引擎；
`RandomSelector<T>` 提供可选 seed 构造参数，供单元测试生成可复现结果。

### 4.3 WEIGHTED_ROUND_ROBIN

每个 `pool_key` 独立维护：

```cpp
struct WeightedState
{
    size_t index = 0;
    uint32_t returned = 0;
};
```

保持现有 HTTP 实现的简单加权轮询语义。例如权重为 `A=2, B=1` 时，首选序列为：

```text
A, A, B, A, A, B ...
```

权重 `0` 归一为 `1`，避免候选因为错误配置产生不明确行为。不可用或已尝试候选会被跳过，并推进到
下一个候选。

### 4.4 LEAST_CONNECTION

每次 `select()` 都调用 `active_requests` 读取最新 active 数，选择最小值。active 相同时按照候选原始
顺序选择第一个，不额外叠加随机或轮询规则，保持算法含义和当前 HTTP 行为简单可预测。

selector 只读取 active 数，不增加或减少 active。调用方负责候选生命周期。

## 5. 并发与生命周期边界

每个 selector 使用自己的 mutex 保护：

- ROUND_ROBIN 的各 pool 下标；
- WEIGHTED_ROUND_ROBIN 的各 pool 权重状态；
- RANDOM 的随机引擎；
- 一次候选过滤与策略选择过程。

访问器会在 selector 临界区内调用。访问器不得回调同一个 selector，也不得长时间阻塞。

### 5.1 HTTP 生命周期

```text
HttpLoadBalanceClient::selectEndpoint()
        ↓
锁住 HttpLoadBalanceClient 的选择临界区
        ↓
CandidateSelector<HttpEndpoint::ptr>::select()
        ↓
HttpEndpoint::beginRequest()
        ↓
释放选择临界区
        ↓
limiter / breaker / connection pool / HTTP
        ↓
所有退出路径 HttpEndpoint::endRequest()
```

`select + beginRequest` 继续在同一个 HTTP client 临界区完成，避免抽取后让 LEAST_CONNECTION
明显退化为多个线程同时看到相同 active 数。

### 5.2 真实 Provider 生命周期

```text
CandidateSelector<ProviderCandidate>::select()
        ↓
ProviderAttemptExecutor 进行 budget / limiter / breaker 检查
        ↓
进入 adapter handler
        ↓
RealProviderRuntime::recordAttemptBegin()
        ↓
Provider HTTP 请求
        ↓
RealProviderRuntime::recordAttemptEnd()
```

真实 Provider 的 `LEAST_CONNECTION` 读取 runtime 中受锁保护的 `ProviderStats::in_flight`。
selector 不预占 Provider active 名额，因此极端并发下允许两个请求同时观察到相同最小值。这是
“selector 只选择、调用方管理生命周期”的明确取舍；本轮不通过公共 selector 接管 Provider 统计来
消除这一近似。

锁顺序保持单向：selector 可以通过 accessor 短暂读取 runtime stats；runtime 的统计和状态输出路径
不反向调用 selector，避免形成锁环。

## 6. HttpLoadBalanceClient 接入

### 6.1 成员调整

保留：

```text
m_endpoints
m_limiter
m_circuitBreaker
m_mutex
```

删除：

```text
m_strategy
m_nextIndex
m_weightedIndex
m_weightedReturned

selectRoundRobin()
selectRandom()
selectWeightedRoundRobin()
selectLeastConnection()
```

新增：

```cpp
sylar::load_balance::CandidateSelector<HttpEndpoint::ptr>::ptr m_selector;
```

### 6.2 HTTP accessors

```text
key             → endpoint->getLimitKey()
available       → endpoint && endpoint->getStatus() == HttpEndpointStatus::UP
weight          → endpoint->getWeight()
active_requests → endpoint->getActiveRequestCount()
```

`HttpLoadBalanceClient::Create()` 使用传入策略和上述 accessors 创建 selector。创建失败时沿用当前
`Create()` 返回空指针的错误边界。

### 6.3 请求内故障转移

`HttpLoadBalanceClient::request()` 继续维护 `tried_endpoint_keys`。每次 endpoint 尝试都重新调用
`selectEndpoint(tried_endpoint_keys)`，因此选择依据是当时最新的 endpoint 状态和 active 数。

限流、熔断或请求失败后，当前 endpoint key 已进入 `tried_endpoint_keys`，本轮不会重复选择。
HTTP retry policy、总尝试次数、连接池和 trace 行为保持不变。

## 7. 真实 Provider 接入

### 7.1 配置加载

删除只接受 ROUND_ROBIN 的 `IsSupportedLoadBalance()`，改为调用公共：

```cpp
ParseLoadBalanceStrategy(parsed.load_balance, &strategy)
```

真实 Provider 接受以下现有名称：

```text
ROUND_ROBIN
RANDOM
WEIGHTED_ROUND_ROBIN
LEAST_CONNECTION
```

未知名称继续在启动配置加载阶段失败，不创建 runtime，也不尝试真实外网请求。

### 7.2 Runtime selector

`RealProviderRuntime` 新增：

```cpp
sylar::load_balance::CandidateSelector<ProviderCandidate>::ptr m_selector;
```

Provider accessors：

```text
key             → candidate.name
available       → candidate.enabled
weight          → candidate.weight
active_requests → runtime.getProviderInFlight(candidate.name)
```

`getProviderInFlight()` 只在 `m_mutex` 保护下读取 `m_stats`，找不到候选时返回 `0`。

当前 `m_requestSequence` 继续用于真实 Provider request id 和 trace，只删除它参与候选列表旋转的职责。
`OrderCandidates()` 被删除。

### 7.3 ProviderAttemptExecutor

新增接收 selector 的构造入口。现有构造函数继续保留，并在未显式传入 selector 时创建默认的
`RoundRobinSelector<ProviderCandidate>`，保证已有测试和调用代码仍可工作。

执行逻辑从固定顺序 `for` 循环调整为动态选择：

```text
candidates = router.findCandidates(request.model)
tried_keys = []

while tried_keys.size() < candidates.size():
    candidate = selector.select(request.model, candidates, tried_keys)
    如果没有候选：结束

    tried_keys 加入 candidate.name
    检查并消耗 RequestExecutionBudget
    尝试 limiter permit
    尝试 breaker guard
    调用 candidate 对应 adapter handler
    分类 Provider 结果并回报 breaker

    不允许切换：立即返回
    允许切换：重新调用 selector
```

预算消耗、limiter、breaker 和 handler 的先后顺序保持当前实现，不借本次 selector 重构调整
“治理拒绝是否消耗尝试预算”等既有语义。

### 7.4 真实 Provider 安全边界

selector 只在 `ProviderAttemptExecutor::shouldTryNextCandidate()` 返回允许切换后再次执行。

保持：

- Provider 401/403 不自动切换；
- Provider 429 可以在共享预算内选择其他候选；
- Provider 5xx 可以在共享预算和非幂等边界内选择其他候选；
- 明确的发送前失败可以选择其他候选；
- `may_have_submitted=true` 且未收到完整响应时禁止再次选择；
- adapter 继续负责每个候选独立的 URL、path、model、Key 和 TLS 配置；
- 对外错误继续使用脱敏后的 gateway error。

## 8. Trace 与状态接口

本轮不改变对外 HTTP body 和 `/internal/status` 的 JSON 主结构。

真实 Provider 请求开始时的候选事件表示“当前逻辑模型的候选池”，实际选择和故障转移顺序以每个
`attempt_result` 事件中的 provider 为准。`buildStatusJson()` 继续输出当前策略名称、各 Provider 的
`in_flight`、成功/失败/限流统计和熔断状态。

状态和 trace 不输出 API Key、Key 环境变量值、完整用户 messages 或 Provider 原始错误体。

## 11. 实施顺序

1. 先为公共 selector 编写失败测试，覆盖四种算法和边界。
2. 实现 `candidate_selector.*`，让纯单元测试通过。
3. 将 `HttpLoadBalanceClient` 接入公共 selector，运行现有 HTTP 负载均衡测试。
4. 为真实 Provider 四种策略和动态故障转移编写失败测试。
5. 将 `ProviderAttemptExecutor` 和 `RealProviderRuntime` 接入公共 selector。
6. 运行真实 Provider 离线测试，确认共享预算、错误分类和不重复提交语义不变。
7. 更新 `文档/真实提供商网关.md` 中的策略支持状态和职责边界。
8. 完成 targeted build、全量 build、unit CTest 和 diff 检查。

所有生产代码都在对应失败测试出现后编写，按 RED、GREEN、REFACTOR 小步推进。

## 12. 完成标准

满足以下条件才视为完成：

1. 两条调用链使用同一套 `CandidateSelector<T>` 算法实现。
2. `HttpLoadBalanceClient` 不再包含四种选择算法的具体实现。
3. 真实 Provider 配置支持四种策略，未知策略启动失败。
4. `ProviderCandidate::weight` 在 WEIGHTED_ROUND_ROBIN 中生效。
5. 真实 Provider LEAST_CONNECTION 使用 runtime 的最新 `in_flight`。
6. 每次故障转移重新选择，并排除本次请求已经尝试的候选。
7. Provider 可能已提交的请求仍不发生自动故障转移。
8. HTTP endpoint 的 `beginRequest/endRequest` 和 Provider 的 `in_flight` 生命周期正确。
9. 新增纯 selector 测试、HTTP 负载均衡测试和真实 Provider 离线测试通过。
10. 默认 Mock/真实 Provider 启用边界、凭据安全和对外响应格式不发生非预期变化。
11. 用户已有的源码格式修改和两份理解文档修改未被覆盖。
12. `git diff --check` 通过，构建产物未混入源码变更。
