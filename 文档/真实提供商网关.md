# 真实提供商网关

> 状态：后续独立工作项。只有 `AI模型调用网关方案草案.md` 的 G0–G5 本地 Mock 闭环、专用测试和故障演示稳定后才开始。本文件不改变离线演示、默认测试或 CI 的边界。

## 1. 目标与边界

本阶段让网关能够调用一个或多个真实的 OpenAI Chat Completions 兼容 Provider，同时继续对外提供 `/v1/chat/completions` 的非流式兼容接口。

它解决的是真实 HTTPS、凭据、供应商错误分类、模型路由和调用治理；不训练模型，也不把某家供应商的 SDK 或协议细节塞进 Servlet。流式 SSE、tools/function calling、token 预算和费用治理另行设计。

真实 Provider 的故障转移只保证网关尽力维持**服务可用性**，不保证两次调用具有完全相同的模型语义、输出文本、usage 或计费结果。只有能力、上下文上限、输出格式、价格和业务语义均可接受的 provider/model 组合，才能进入同一逻辑模型池。

## 2. 前置条件

开始前必须满足：

1. 已完成本地 Mock 网关 G0–G5；真实 endpoint 默认关闭，不能替代本地测试夹具。
2. 已完成客户端 TLS 安全默认值：可信 CA、主机名校验和 SNI；不能为了连通而关闭证书校验。
3. 网关协议层统一使用 jsoncpp；不以字符串拼接解析或生成外部 JSON。
4. API Key 只通过环境变量名或本地忽略配置引用；不得写入仓库、日志、状态接口、错误响应或测试快照。
5. 默认测试、CI 和一键 Mock 演示都不依赖外网、真实 Key 或真实计费账户。

## 3. 组件边界

```text
调用方（OpenAI-compatible 请求）
             |
             v
AiGatewayServlet / 网关协议层（jsoncpp）
             |
             v
逻辑模型路由 + 业务请求预算
             |
             v
AiProviderAdapter
             |
             v
HttpLoadBalanceClient + limiter + breaker
             |
             v
OpenAICompatibleProvider / 其他专用适配器
```

- `AiGatewayServlet`：验证 OpenAI 兼容请求，建立一次业务请求上下文，返回兼容响应或错误对象；不直接拼装某个 provider 的请求。
- 协议类型：定义内部 `GatewayChatRequest`、`GatewayChatResponse` 与统一错误分类；只依赖 jsoncpp，不管理连接或 endpoint。
- 逻辑模型路由：把外部 `model` 映射到一组兼容的 provider/model；不把任意模型随机混进一个负载均衡池。
- `AiProviderAdapter`：将内部请求转换为特定 provider 的 URL、认证头和 JSON，并解析响应；不实现负载均衡、限流或熔断状态机。
- HTTP 治理链：继续由 `HttpLoadBalanceClient`、`HttpConcurrencyLimiter`、`HttpCircuitBreaker` 负责候选选择、准入和后端失败统计。

第一个适配器为 `OpenAICompatibleProvider`，通过 base URL、chat path、上游模型名和 API Key 环境变量工作。只有认证、请求或响应形状确实不兼容时，才增加专用 adapter；不为调用某一家 Provider 引入其 C++ SDK。

`mock_model_provider` 应逐步复用同一内部协议/适配器契约，使真实 Provider 的大部分转换与错误分类可由本地契约测试覆盖。

## 4. 逻辑模型与故障转移语义

配置先定义逻辑模型，例如 `general-chat`，再为其配置可互换的候选：

```text
调用方请求 general-chat
        ↓
逻辑模型路由表
        ↓
general-chat 的兼容 provider/model 池
        ↓
HttpLoadBalanceClient 选择 endpoint
```

同一池的自动切换表示“尽可能完成本次服务调用”，不是“保证得到同一答案”。如果候选模型的能力、版本、价格或内容策略不同，应作为显式降级策略并向调用方/配置清楚声明，不能在失败后悄悄随机替换。

### 已发送请求的超时与重复计费

下游请求在连接失败、写入前失败等能够明确确认“尚未发送”的情况下，可以在共享预算内重试或切换。若请求可能已经被 Provider 接收，但网关因读超时、连接中断等未得到响应，则默认**不自动重试或故障转移**：再次发送可能产生两次模型执行、两次计费和两份不同内容。

只有同时满足以下条件，才可以把“已发送后未知结果”的请求纳入自动重试：

1. 目标 Provider 明确支持本接口的幂等键或等价去重契约；
2. 网关为同一业务请求在所有重试中稳定传递该键，并能确认 Provider 的去重语义覆盖计费与执行；
3. 该操作在业务上允许延迟返回旧结果或重复读取同一结果。

否则网关应返回一个可区分的超时/未知结果错误，并记录“可能已提交”状态供调用方按其业务语义处理；不能把它伪装成安全的失败重试。

## 5. 总 deadline 与总尝试预算

每个入站请求创建一个不可重置的 `RequestExecutionBudget`：

- **总 deadline**：从网关收到请求时开始计时，后续单实例 I/O、重试、候选切换和适配器处理都只能使用剩余时间；不得在每次切换时重新给满超时。
- **总尝试预算**：首次下游调用、`HttpClient` 的单实例重试和 `HttpLoadBalanceClient` 的 endpoint 故障转移统一计数；每次真正准备发起下游 HTTP 尝试前扣减一次。
- **耗尽行为**：剩余时间或尝试次数为零时立即停止新的下游调用，释放已持有资源，并返回明确的超时或上游不可用错误。

这项预算必须由网关请求上下文统一传递给 adapter、`HttpClient` 和负载均衡路径，而不是让各层各自维护 `max_retry`。它防止“每个 endpoint 重试 N 次，再故障转移 M 个 endpoint”叠加为请求风暴。

配置表达意图如下：

```yaml
real_providers:
  request_deadline_ms: 15000
  max_total_attempts: 3
  providers:
    - name: provider-a-chat
      enabled: false
      type: openai_compatible
      logical_model: general-chat
      base_url: https://api.provider.example
      chat_path: /v1/chat/completions
      upstream_model: provider-a-model
      api_key_env: PROVIDER_A_API_KEY
```

`request_deadline_ms` 和 `max_total_attempts` 是一项业务请求的总量，不是每个 endpoint 的独立额度。单个 HTTP 请求的 connect/read 超时也必须被剩余 deadline 截断。

## 6. 错误分类、限流和熔断

| 情况 | 网关动作 | 是否记入 breaker |
| --- | --- | --- |
| 调用方请求不合法、模型不存在（通常为 4xx） | 直接返回兼容参数错误，不重试 | 否 |
| API Key 缺失、失效或权限不足（401/403） | 返回不泄密的认证/配置错误并告警 | 否 |
| Provider 429 | 不无限重试；可在总预算内尝试同一逻辑池的其他候选 | 否，单独记录 quota 状态 |
| 发送前网络失败、连接超时、Provider 5xx | 在总 budget 内重试或故障转移 | 是 |
| 请求可能已发送后的读超时/未知结果 | 默认不重试，返回可区分错误 | 按连接/服务失败指标记录，但不据此重复提交请求 |

outbound limiter 仍是建立下游请求前的有界准入：当前候选满额时尝试其他候选；所有候选均不可准入时快速失败，绝不无限排队。所有返回路径都必须释放已获取的 limiter permit，令 in-flight 计数回到正确基线。

breaker 进入 `HALF_OPEN` 后，只允许少量探测请求（`half_open_max_requests`，推荐默认 `1`）通过；在这些探测成功并达到恢复条件前，不能突然恢复全部流量。探测失败应立即重新打开，避免恢复瞬间再次把不稳定 Provider 打爆。

## 7. 凭据、日志与观测

- 没有对应环境变量时，启动阶段给出明确但不泄密的配置错误，不尝试对外连接。
- 日志只记录内部 request id、provider 名、耗时、HTTP 类别、已用/剩余尝试次数、deadline 是否耗尽、限流和熔断状态；不记录 `Authorization`、API Key、完整 messages 或完整模型响应。
- `/internal/status` 只展示脱敏后的 provider 名和运行状态，不能回显敏感查询参数、完整 base URL 或环境变量内容。
- 并发/QPS limiter 是第一层保护；token/分钟估算、日预算、计费对账和用户配额属于后续独立能力，不能在本阶段假装解决。

## 8. 实施切片与验证

| 切片 | 修改范围 | 独立验证 |
| --- | --- | --- |
| R0 | TLS 安全默认值与 jsoncpp 接入 | 受信 CA 成功；错误主机名、未信任证书失败；协议层 JSON 契约测试 |
| R1 | 统一请求/响应类型、逻辑模型路由和 `AiProviderAdapter` | Mock 的正常、超时、错误用例保持通过；不兼容模型不会进入同一池 |
| R2 | `OpenAICompatibleProvider` 非流式适配器 | 本地录制响应或假 HTTPS 服务验证请求、认证头和响应解析 |
| R3 | 共享 deadline/总尝试预算与错误分类 | 断言 retry + failover 的下游尝试总数不超过预算；任何尝试不能超过剩余 deadline |
| R4 | limiter、breaker 与语义安全回归 | 成功、失败、超时、异常、故障转移后 permit 都释放且 in-flight 无泄漏；半开只放行配置数量的探测请求 |
| R5 | 手工真实 smoke test | 显式设置 Key 后只发起一次受控请求；不进入默认测试或 CI |

R3 必须额外覆盖“发送前失败可切换”和“请求可能已发送后超时默认不重试”两类测试。R4 的 permit/in-flight 断言必须在每一种退出路径结束后读取计数，不能只检查最终成功响应。

## 9. 明确不做

- 不让真实 Provider 的访问阻塞本地 Mock 网关交付。
- 不承诺故障转移后的输出、usage 或计费与原 Provider 完全一致。
- 不对可能已发送的非幂等模型调用做盲目重试。
- 不做无界队列、无限重试，也不让各层独立重置 budget。
- 不做流式 SSE、tools/function calling、视觉输入、token 计费治理或用户账单系统。
