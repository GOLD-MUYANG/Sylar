# Sylar 基础封装速查

这 7 个文件整体上确实属于基础封装层，但它们封装的对象并不一样：有的直接包装
Linux/POSIX 接口，有的负责把底层对象组织成更容易被 HTTP、WebSocket 和模块系统使用的
C++ 抽象。

阅读这部分时不需要逐行看实现，只要能回答三个问题：

1. 它封装了什么对象？
2. 它最终落到了哪些 Linux/POSIX 接口？
3. 上层为什么不直接使用这些底层接口？

## 1. 先看整体分层

```text
地址表示与解析
Address
  -> sockaddr / sockaddr_in / sockaddr_in6 / sockaddr_un

文件描述符运行状态
FdManager
  -> fd 是否为 socket、非阻塞状态、收发超时
  -> 为 Hook + IOManager 提供判断依据

内存与流式 I/O
ByteArray -> Stream -> SocketStream -> Socket
                                      -> send / recv
                                      -> sendmsg / recvmsg
                                      -> close

动态模块
Library
  -> dlopen / dlsym / dlclose

数据工具
HashUtil
  -> hash / Base64 / MD5 / SHA / HMAC / 十六进制 / 字符串工具
```

一句话概括：

> 这些文件把 C 风格地址、裸文件描述符、分块内存、socket 收发、动态库句柄和散落的
> 编解码算法，整理成框架内部可以复用的 C++ 对象和接口。

## 2. 七个文件总表

| 文件 | 主要封装对象 | 直接或间接接触的底层接口 | 上层得到什么 |
| --- | --- | --- | --- |
| `sylar/address.cc` | IPv4、IPv6、Unix 域和未知类型地址 | `getaddrinfo`、`freeaddrinfo`、`getifaddrs`、`freeifaddrs`、`inet_pton` | 不再直接操作各种 `sockaddr_*` 结构 |
| `sylar/fd_manager.cc` | fd 的运行状态和附加元数据 | `fstat`、`fcntl`，以及 `O_NONBLOCK`、`SO_RCVTIMEO`、`SO_SNDTIMEO` | Hook 能判断某次 I/O 应该直接执行还是转交协程调度 |
| `sylar/bytearray.cc` | 可扩容的分块二进制缓冲区 | `iovec` 数据结构；文件部分使用 C++ 文件流 | 统一的序列化、反序列化、游标和分散缓冲区能力 |
| `sylar/stream.cc` | “读、写、关闭”的抽象接口 | 不直接调用系统接口 | 屏蔽一次 I/O 可能只完成部分数据的问题 |
| `sylar/socket_stream.cc` | 把 `Socket` 适配成 `Stream` | 间接使用 `send`、`recv`、`sendmsg`、`recvmsg`、`close` | HTTP、WebSocket 可以按流来读写 socket |
| `sylar/library.cc` | `.so` 动态库和 `Module` 生命周期 | `dlopen`、`dlsym`、`dlclose`、`dlerror` | 模块可以在运行时被发现、创建和卸载 |
| `sylar/util/hash_util.cc` | Hash、摘要、编码和字符串工具 | OpenSSL 的 MD5/SHA 接口；不直接调用 Linux 系统调用 | 调用方不需要重复实现常用数据处理逻辑 |

## 3. `address.cc`：封装网络地址

### 3.1 封装了什么

Linux socket 接口通常通过 `sockaddr*` 接收地址，但实际地址结构会随地址族变化：

| Sylar 对象 | 底层结构 | 表示内容 |
| --- | --- | --- |
| `IPv4Address` | `sockaddr_in` | IPv4 地址和端口 |
| `IPv6Address` | `sockaddr_in6` | IPv6 地址和端口 |
| `UnixAddress` | `sockaddr_un` | Unix 域 socket 路径 |
| `UnknownAddress` | `sockaddr` | 框架暂时不认识的地址族 |

`Address` 提供统一基类，上层只需要拿到 `Address::ptr`，需要调用 socket 接口时再通过
`getAddr()` 和 `getAddrLen()` 取出底层结构。

它还封装了以下能力：

- 解析 `host:port` 和 `[IPv6]:port`。
- 把域名解析成一个或多个地址对象。
- 枚举本机网卡、网卡地址和前缀长度。
- 计算广播地址、网络地址和子网掩码。
- 统一输出 IPv4、IPv6 和 Unix 域地址字符串。
- 在主机字节序与网络字节序之间转换端口和地址字段。

### 3.2 用到的 Linux/POSIX 接口

| 接口 | 在这里解决的问题 |
| --- | --- |
| `getaddrinfo` | 根据域名、IP、端口、地址族和 socket 类型生成地址链表 |
| `freeaddrinfo` | 释放 `getaddrinfo` 返回的地址链表 |
| `getifaddrs` | 获取本机所有网卡及其地址、掩码信息 |
| `freeifaddrs` | 释放 `getifaddrs` 返回的链表 |
| `inet_pton` | 把 IPv4/IPv6 字符串转成二进制网络地址 |
| `ntohs` | 在输出 IPv6 地址时把网络字节序端口转回主机字节序 |
| `byteswapOnLittleEndian` | 项目自己的字节序辅助函数，IPv4/IPv6 的端口存取主要通过它转换 |

严格来说，`getaddrinfo`、`getifaddrs`、`inet_pton` 属于 libc/POSIX 网络接口，不是直接的
Linux 内核系统调用；但日常讲项目时，可以把它们归到 Linux 网络编程接口这一层。

### 3.3 在项目里的位置

```text
配置里的监听地址 / HTTP 客户端的 host
-> Address::Lookup / LookupAnyIPAddress
-> IPv4Address / IPv6Address / UnixAddress
-> Socket::bind / Socket::connect
```

可以这样讲：

> `Address` 把 Linux 的多套 `sockaddr` 结构统一成面向对象的地址体系，同时封装域名解析、
> 网卡枚举和子网计算，让服务器绑定和客户端连接都使用同一套地址接口。

## 4. `fd_manager.cc`：给裸 fd 补充运行状态

### 4.1 封装了什么

Linux 的 fd 本质上只是一个整数。只看这个整数，框架不知道：

- 它是不是 socket。
- 它是否已经初始化；结构中也预留了关闭状态。
- 内核层面是否已被框架设置成非阻塞。
- 用户是否主动要求非阻塞。
- socket 的接收、发送超时是多少。

`FdCtx` 保存单个 fd 的这些附加状态，`FdManager` 用一个按 fd 下标访问的 `vector` 统一管理
`FdCtx`，并用读写锁保护并发访问。

这里最重要的边界是：

> `FdManager` 不是文件对象，也不是连接池；它是 fd 到运行时元数据的映射表。

### 4.2 用到的 Linux/POSIX 接口

| 接口或常量 | 在这里解决的问题 |
| --- | --- |
| `fstat(fd, &stat)` | 判断 fd 是否有效，并取得 fd 对应对象的类型信息 |
| `S_ISSOCK(stat.st_mode)` | 判断 fd 是否对应 socket |
| `fcntl(fd, F_GETFL)` | 读取 fd 当前状态标志 |
| `fcntl(fd, F_SETFL, flags \| O_NONBLOCK)` | 把 socket 在内核层设置成非阻塞 |
| `SO_RCVTIMEO` | 标识接收超时，框架把超时毫秒数记录到 `FdCtx` |
| `SO_SNDTIMEO` | 标识发送超时，框架把超时毫秒数记录到 `FdCtx` |

源码里使用的是 `fcntl_f`，它是 Hook 初始化时通过 `dlsym(RTLD_NEXT, "fcntl")` 保存的原始
`fcntl` 函数指针。这样初始化 fd 状态时可以绕过 Sylar 自己的 `fcntl` Hook 包装。

### 4.3 为什么要同时记录两种非阻塞状态

| 状态 | 含义 |
| --- | --- |
| `m_sysNonblock` | Sylar 为了接入 epoll 和协程调度，在内核 fd 上强制设置的非阻塞状态 |
| `m_userNonblock` | 业务代码是否主动通过 `fcntl` 或 `ioctl` 要求非阻塞 |

如果用户没有主动要求非阻塞，Hook 可以把一次看起来会阻塞的 socket I/O 转换成：先尝试
非阻塞调用，遇到 `EAGAIN` 就把当前协程挂起，并交给 `IOManager` 等待事件。如果用户明确
要求非阻塞，Hook 则尊重用户语义，直接返回系统调用结果。

```text
socket / accept 创建 fd
-> FdManager 建立 FdCtx
-> fstat 判断是不是 socket
-> fcntl 设置内核非阻塞
-> Hook 查询非阻塞状态和超时
-> 决定直接返回，还是交给 IOManager 等待
```

可以这样讲：

> `FdManager` 给裸 fd 增加 socket 类型、非阻塞语义和超时等上下文，是 Hook 把阻塞式
> socket API 改造成协程友好 I/O 的状态基础。

## 5. `bytearray.cc`：封装二进制数据和分块缓冲区

### 5.1 封装了什么

`ByteArray` 不是简单的 `char*`，而是由固定大小 `Node` 串成的链式缓冲区。它统一维护：

- 当前读写位置 `m_position`。
- 已写入数据大小 `m_size`。
- 总容量和按需扩容。
- 固定长度整数、浮点数和字符串的读写。
- ZigZag + Varint 变长整数编码，减少小整数占用空间。
- 大端、小端字节序转换。
- 二进制数据与字符串、十六进制文本、文件之间的转换。
- 把不连续内存导出成多个 `iovec`，供 scatter/gather I/O 使用。

### 5.2 和 Linux 接口的关系

`ByteArray` 本身不直接执行 `read(2)`、`write(2)`、`readv(2)` 或 `writev(2)`。它主要使用
`memcpy` 操作内存，使用 C++ `ifstream` / `ofstream` 读写文件。

它与 Linux 网络 I/O 的连接点是 `iovec`：

```text
ByteArray 的多个内存块
-> getReadBuffers / getWriteBuffers
-> vector<iovec>
-> SocketStream
-> Socket::send(iovec*) / Socket::recv(iovec*)
-> sendmsg / recvmsg
```

`iovec` 只是 Linux/POSIX 用来描述多段内存的结构，不是一次系统调用。真正执行收发的是
后面的 `sendmsg` / `recvmsg`。

可以这样讲：

> `ByteArray` 把分块内存、读写游标、基础类型序列化和字节序处理放到一个对象中，并能把
> 多个内存块导出为 `iovec`，避免上层为了 socket 收发先把数据拼成一整块连续内存。

## 6. `stream.cc`：抽象统一的流式读写

### 6.1 封装了什么

`Stream` 定义了最小的流接口：

```text
read
write
close
```

每个 `read` / `write` 又同时支持裸内存和 `ByteArray`。具体数据来自 socket、文件还是其他
对象，由子类决定。

`Stream` 自己真正实现的是 `readFixSize` 和 `writeFixSize`：普通 I/O 一次可能只完成部分
数据，这两个方法会循环调用底层 `read` / `write`，直到完成指定长度，或者遇到错误、连接
关闭。

### 6.2 和 Linux 接口的关系

`Stream` 是纯 C++ 抽象层，不直接调用 Linux 接口。它解决的是系统 I/O 常见的“短读、短写”
语义，而不是负责具体 fd。

可以这样讲：

> `Stream` 把不同 I/O 对象统一成读、写、关闭接口，并在这一层集中处理短读短写，上层协议
> 不需要反复编写“还差多少字节”的循环。

## 7. `socket_stream.cc`：把 Socket 适配成 Stream

### 7.1 封装了什么

`SocketStream` 同时持有一个 `Socket` 和一个 `owner` 标记：

- `owner = true`：`SocketStream` 析构时关闭 socket。
- `owner = false`：只借用 socket，不负责其生命周期。
- 裸内存读写直接委托给 `Socket::recv` / `Socket::send`。
- `ByteArray` 读写先生成 `iovec` 数组，再委托给 `Socket`。
- I/O 成功后推进 `ByteArray` 游标。

它只负责“已经建立的 socket 怎么按流读写”，不负责 `socket`、`bind`、`listen`、`accept`
和 `connect` 这些建连过程。

### 7.2 最终落到哪些 Linux 接口

| `SocketStream` 路径 | `Socket` 最终调用 |
| --- | --- |
| `read(void*, length)` | `recv` |
| `write(void*, length)` | `send` |
| `read(ByteArray, length)` | `recvmsg` |
| `write(ByteArray, length)` | `sendmsg` |
| `close()` 或拥有者析构 | `close` |

这些调用还会经过 Sylar 的 Hook。开启 Hook 时，如果 socket 暂时不可读或不可写，当前协程
可以让出执行权，由 `IOManager` 等待 epoll 事件后再继续。

```text
HttpSession / HttpConnection / WSSession
-> Stream::readFixSize / writeFixSize
-> SocketStream::read / write
-> Socket::recv / send / recvmsg / sendmsg
-> Hook
-> IOManager + epoll
```

可以这样讲：

> `SocketStream` 是 `Socket` 到通用 `Stream` 接口的适配器，让 HTTP 和 WebSocket 只关心
> 流式读写，同时复用 Socket、Hook 和 IOManager 提供的底层网络能力。

## 8. `library.cc`：封装动态库模块加载

### 8.1 封装了什么

`Library::GetModule(path)` 把一个 `.so` 文件转换成受 `shared_ptr` 管理的 `Module`：

```text
模块 .so 路径
-> dlopen 打开动态库
-> dlsym 查找 CreateModule
-> dlsym 查找 DestoryModule
-> CreateModule 创建模块对象
-> shared_ptr + ModuleCloser 管理生命周期
-> DestoryModule 销毁模块对象
-> dlclose 关闭动态库
```

加载完成后，它还会重新扫描配置目录，使模块注册的配置项能够立即接收配置文件中的值。

### 8.2 用到的 Linux 动态链接接口

| 接口 | 在这里解决的问题 |
| --- | --- |
| `dlopen(path, RTLD_NOW)` | 打开 `.so`，并立即解析需要的符号 |
| `dlsym(handle, name)` | 从动态库中查找 `CreateModule`、`DestoryModule` |
| `dlerror()` | 获取最近一次动态链接操作的错误信息 |
| `dlclose(handle)` | 模块对象销毁后关闭动态库 |

这些接口来自 `libdl`，属于 Linux/Unix 动态链接 API，不是直接的内核系统调用。

可以这样讲：

> `Library` 约定每个模块动态库导出创建和销毁函数，再用 `dlopen`、`dlsym` 和自定义
> `shared_ptr` 删除器把动态库句柄与模块对象生命周期绑定起来。

## 9. `hash_util.cc`：封装数据处理工具

### 9.1 封装了什么

这个文件名叫 `hash_util`，实际包含的能力比 Hash 更广：

| 类别 | 能力 |
| --- | --- |
| 非加密 Hash | MurmurHash3 32/64 位、`quick_hash` |
| 编解码 | Base64 编码和解码 |
| 摘要 | MD5、SHA-1，支持二进制结果和十六进制结果 |
| 消息认证 | HMAC-MD5、HMAC-SHA1、HMAC-SHA256 |
| 格式转换 | 二进制与十六进制字符串互转 |
| 字符串工具 | `replace`、`split` |
| 随机字符串 | 从数字和大小写字母中随机选取字符 |

MD5 和 SHA 相关实现调用 OpenSSL 的 `MD5_Init/Update/Final`、`SHA1_Init/Update/Final`
等接口。Base64、MurmurHash、十六进制转换和字符串处理则是在文件内自行实现。

### 9.2 和 Linux 接口的关系

这里没有直接调用 Linux 系统调用。它属于算法和第三方库封装层，不属于操作系统资源封装。

需要注意：`random_string` 当前基于 C 库 `rand()`，适合生成普通随机文本，不应该描述成
密码学安全随机数。当前 `sha0sum` 的实现也调用了 `SHA1_*`，讲项目时按源码事实说明即可，
不要把它扩展成独立实现的 SHA-0 算法。

可以这样讲：

> `HashUtil` 集中提供框架常用的 Hash、Base64、摘要、HMAC 和字符串转换能力，其中摘要
> 算法复用 OpenSSL；它是数据工具集合，不是 Linux 系统调用封装。

## 10. Linux/POSIX 接口集中速查

### 10.1 直接出现在这 7 个实现中的接口

| 接口 | 所在文件 | 作用 |
| --- | --- | --- |
| `getaddrinfo` / `freeaddrinfo` | `address.cc` | 域名和服务解析、地址链表释放 |
| `getifaddrs` / `freeifaddrs` | `address.cc` | 本机网卡地址枚举、结果释放 |
| `inet_pton` | `address.cc` | 文本 IP 转二进制地址 |
| `ntohs` | `address.cc` | 输出 IPv6 地址时转换端口字节序 |
| `byteswapOnLittleEndian` | `address.cc` | 项目内部完成 IPv4/IPv6 地址和端口的字节序转换 |
| `fstat` | `fd_manager.cc` | 检查 fd 并识别对象类型 |
| `fcntl` | `fd_manager.cc` | 读取和设置 `O_NONBLOCK`；源码通过原始函数指针调用 |
| `dlopen` / `dlsym` / `dlclose` / `dlerror` | `library.cc` | 动态库打开、符号查找、关闭和错误获取 |

### 10.2 通过 `Socket` 间接使用的接口

| 接口 | 从哪个封装进入 | 作用 |
| --- | --- | --- |
| `recv` | `SocketStream::read(void*)` | 从 socket 读入连续内存 |
| `send` | `SocketStream::write(void*)` | 把连续内存写入 socket |
| `recvmsg` | `SocketStream::read(ByteArray)` | 把数据分散写入多个 `iovec` 缓冲块 |
| `sendmsg` | `SocketStream::write(ByteArray)` | 从多个 `iovec` 缓冲块聚集发送 |
| `close` | `SocketStream::close` / 拥有者析构 | 关闭 socket fd |

### 10.3 容易误认为“方法”的底层类型和常量

| 名称 | 实际是什么 |
| --- | --- |
| `sockaddr_in` / `sockaddr_in6` / `sockaddr_un` | 不同地址族使用的 C 结构体 |
| `iovec` | 描述一段内存地址和长度的结构体 |
| `AF_INET` / `AF_INET6` / `AF_UNIX` | 地址族常量 |
| `O_NONBLOCK` | fd 非阻塞状态标志 |
| `SO_RCVTIMEO` / `SO_SNDTIMEO` | socket 接收和发送超时选项名 |
| `RTLD_NOW` | `dlopen` 的符号解析策略 |

## 11. 最后只记这一段

如果只准备讲 30 秒，可以直接说：

> 这一组代码是 Sylar 的通用基础封装。`Address` 统一 IPv4、IPv6 和 Unix 域地址，并包装
> 域名解析和网卡枚举；`FdManager` 给裸 fd 记录 socket、非阻塞和超时状态，为 Hook 和
> 协程 I/O 提供判断依据；`ByteArray` 管理分块二进制缓冲和序列化，`Stream` 处理统一读写
> 以及短读短写，`SocketStream` 再把这套流接口接到 `send/recv/sendmsg/recvmsg`；`Library`
> 用 `dlopen/dlsym/dlclose` 管理动态模块；`HashUtil` 集中提供 Hash、Base64、摘要和 HMAC。
> 所以它们都是封装，但只有地址、fd、socket 流和动态库这几部分真正接触 Linux/POSIX
> 接口，另外几部分主要是 C++ 抽象和数据工具。
