README 现在应该当“地图”，但不要按 README 从日志系统一路线性读到最后。你应该从**最能串起项目价值的主链路**开始整理。

**推荐顺序**
1. **先整理项目总览**
从 README 开头的扩展概览开始：这个项目现在不是单纯 Sylar 基础框架，而是：

```text
Linux C++ 协程/网络框架
-> HTTP 客户端治理链路
-> AI 模型调用网关
-> 真实 Provider 可选演示路径
```

总纲：  
“这个项目以 Sylar 原有协程网络框架为基础，向上扩展 HTTP 客户端治理链路，并最终落到 AI 模型调用网关 demo。”

2. **再整理第一条主线：服务端运行链路**

先看这条：

```text
main
-> Application
-> Env / Config / Daemon
-> Worker
-> Module
-> TcpServer
-> HttpServer
-> Servlet
```

对应文件：

```text
sylar/main.cc
sylar/application.cc
sylar/env.cc
sylar/config.cc
sylar/daemon.cc
sylar/worker.cc
sylar/module.cc
sylar/tcp_server.cc
sylar/http/http_server.cc
sylar/http/servlet.cc
```

这一部分回答的是：**一个服务怎么启动、怎么读取配置、怎么加载模块、怎么监听端口、怎么处理请求。**

3. **然后整理第二条主线：协程网络底座**

再看：

```text
Fiber
-> Scheduler
-> IOManager / epoll
-> Timer
-> Hook
-> Socket
```

对应文件：

```text
sylar/fiber.cc
sylar/scheduler.cc
sylar/iomanager.cc
sylar/timer.cc
sylar/hook.cc
sylar/socket.cc
```

这一部分回答的是：**为什么这个框架能用同步写法处理异步 IO。**

4. **再整理第三条主线：HTTP 客户端治理链路**

这是现在最适合写进简历和讲项目的链路：

```text
HttpConnectionPool

-> HttpClient
-> retry
-> HttpLoadBalanceClient
-> HttpConcurrencyLimiter
-> HttpCircuitBreaker
```

对应文件：

```text

sylar/http/http_load_balance_client.cc

```

这一部分回答的是：**项目从普通 HTTP 请求，扩展到了连接池、超时、重试、负载均衡、限流、熔断。**

5. **然后整理第四条主线：AI 模型调用网关**

这就是扩展能力的落地点：

```text

```

对应文件：

```text

```

这一部分回答的是：**HTTP 客户端治理链路具体用到了哪里。**

6. **最后整理第五条主线：真实 Provider 路径**

这一块不要和 Mock 网关混成一句话。单独整理：

```text

-> OpenAICompatibleProviderAdapter

```

对应文件：

```text

```

这一部分回答的是：**真实 Provider 能力已经做了基础链路，但默认关闭，需要 API Key 手工验证。**

7. **主线之后补充基础支撑部分**

前面几条主线先解决“项目怎么跑起来、HTTP 客户端治理能力怎么形成、AI 网关怎么落地”的问题。
这些基础文件不一定适合放在第一轮主线里讲，但第二轮整理时需要补上，否则对框架底层能力和 HTTP 协议栈的理解会不完整。

第一块是通用基础工具层：

```text
Address
-> FdManager
-> ByteArray

-> SocketStream
-> Library
-> HashUtil
```

对应文件：

```text

```

这一部分回答的是：**框架内部怎么抽象地址、文件描述符、二进制缓冲、流式读写、socket 流、动态库加载和 hash 工具。**

第二块是 HTTP 解析和 WebSocket 扩展层：

```text
HttpParser
-> HTTP/1.1 parser
-> HTTP client parser
-> WebSocket Connection / Session / Server / Servlet
```

对应文件：

```text

```

这一部分回答的是：**HTTP 报文如何被解析成请求/响应对象，以及 WebSocket 如何复用 HTTP 连接体系扩展出长连接通信能力。**

**你可以按这个节奏整理**
第一轮只整理“项目地图”，不要钻细节：

```text
1. 项目解决什么问题
2. 原始 Sylar 基础能力
3. 我扩展了哪些 HTTP 客户端能力

```

```text
每个模块只写四件事：

```

```text

```
