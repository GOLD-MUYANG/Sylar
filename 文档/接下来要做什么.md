高并发 HTTP 能力建设路线

一、当前项目已经具备的基础

1. 高并发 HTTP 服务端基础

当前项目已经有 IOManager、Scheduler、Fiber、Hook、TcpServer、HttpServer 这一套基础。

IOManager 使用 epoll 管理 IO 事件，Hook 把 socket 读写接入协程调度，HttpServer 基于 TcpServer 完成 HTTP 请求解析、Servlet 分发和响应发送。

文档/压力测试.md 中已经记录过服务端压测结果：

- 短连接：ab -n 50000 -c 100，约 5448 QPS
- 长连接：ab -k -n 50000 -c 100，约 23650 QPS

所以“高并发 HTTP 服务端底座”已经有了。

2. HTTP 客户端基础

sylar/http/http_connection.h 中已经有 HttpConnection：

- DoGet()
- DoPost()
- DoRequest()
- recvResponse()
- sendRequest()

这些接口可以完成基础 HTTP 客户端请求，并且支持 HTTP 和 HTTPS。

3. HttpConnectionPool 第一阶段已修复

项目已经有 HttpConnectionPool：

- getConnection()
- doGet()
- doPost()
- doRequest()
- max_size
- max_alive_time
- max_request

并且已经完成第一阶段修复，可以作为后续 RPC/HTTP Client 的基础连接池继续建设。

本次已完成：

- 新建连接时设置 m_createTime。
- 修正 max_alive_time 判断，过期连接不再复用。
- 实现 m_maxSize 最大连接数限制。
- 连接池满时支持协程等待。
- 等待连接支持超时，超时后返回 POOL_GET_CONNECTION。
- 连接归还时统一判断是否还可复用，包括断开、过期、超过 max_request。
- 响应解析同步 Connection: close / keep-alive 状态，便于连接池决定是否复用连接。
- test_http_connection 改为本地测试，不再依赖外网。

已验证：

- cmake --build build 通过。
- ./bin/test_http_connection 通过。

仍需注意：

- 当前连接池等待超时暂时复用 doRequest(req, timeout_ms) 里的 timeout_ms，后续需要和完整 HTTP 客户端超时模型统一。
- 连续复用同一个 HTTP 连接发多次请求时，响应读取路径仍有进一步梳理空间，后续封装 HttpClient 时一并处理更合适。

4. 超时控制已有一部分

Hook 层已经有 connect/read/write 的协程超时机制。

HttpConnection 和 HttpConnectionPool 里也会设置 socket recv timeout。

但目前还不是完整的 HTTP 客户端超时模型。

后续最好统一成：

- connect_timeout：连接超时
- send_timeout：发送超时
- recv_timeout：读取响应超时
- total_timeout：一次请求总超时

5. Worker 和并发限制材料

项目里已经有 WorkerGroup 和 FiberSemaphore，可以作为“并发数限制”的基础。

但 WorkerGroup 当前没有在实际 HTTP/RPC 请求链路中接入，而且构造函数里没有把 FiberSemaphore 初始化为 batch_size，直接使用会有卡住风险。

6. 简单 worker 负载分摊

WorkerManager::get() 对同名 Scheduler 会随机返回一个，可以算是 worker 内部的简单负载分摊。

但这不是多实例 HTTP 服务负载均衡。

二、当前还没有真正实现的能力

1. 失败重试

当前没有 HTTP/RPC 层面的失败重试。

hook.cc 里的 retry 只是系统调用层面对 EINTR/EAGAIN 的处理，不是业务请求重试。

2. 限流保护

当前没有接入 HTTP/RPC 请求链路的限流器。

可以复用 FiberSemaphore 思路做并发限制，但还没有 token bucket、漏桶、QPS 限制等能力。

3. 熔断降级

当前没有 CircuitBreaker。

还缺少这些能力：

- 失败率统计
- 连续失败统计
- 熔断打开
- 半开探测
- 熔断恢复
- 降级回调或默认响应

4. 多实例负载均衡

当前没有真正面向多个服务实例的负载均衡。

后续需要抽象 Endpoint，并让每个 Endpoint 拥有自己的连接池，然后在请求时做实例选择。

三、推荐建设顺序

推荐不要直接从熔断、限流、负载均衡开始。

这些能力都依赖稳定的客户端请求链路：

HTTP Client -> 连接池 -> 超时控制 -> 错误分类 -> 重试 -> 负载均衡 -> 限流/熔断

建议按下面顺序做。

第一步：先修 HttpConnectionPool（已完成）

这是最优先的事情。

已完成的内容：

- 新建连接时设置 m_createTime。
- 修正 max_alive_time 判断。
- 真正实现 m_maxSize 最大连接数限制。
- 连接池满时支持等待。
- 等待连接支持超时。
- 连接归还时判断连接是否还可复用。
- 增加本地测试，不依赖外网。

这一步完成后，后面的 RPC Client、HTTP Client、重试、负载均衡才有可靠基础。

第二步：统一 HTTP 客户端超时模型（下一步）

新增类似 HttpRequestOptions 的参数对象。

建议字段：

- connect_timeout_ms
- send_timeout_ms
- recv_timeout_ms
- total_timeout_ms

然后让 HttpConnection 和 HttpConnectionPool 都使用同一套参数。

第三步：封装稳定的 HttpClient

不要让业务层直接操作 HttpConnectionPool。

新增一个更上层的 HttpClient，例如：

- HttpClient::request()
- HttpClient::get()
- HttpClient::post()

HttpClient 内部管理连接池、超时、错误码和日志。

同时补充清晰的错误分类：

- INVALID_URL
- INVALID_HOST
- CONNECT_FAIL
- CONNECT_TIMEOUT
- SEND_FAIL
- SEND_TIMEOUT
- RECV_TIMEOUT
- RESPONSE_PARSE_FAIL
- HTTP_STATUS_ERROR

第四步：实现失败重试

重试应该建立在错误分类之后。

建议规则：

- GET、HEAD 等幂等请求默认允许重试。
- POST/RPC 默认不自动重试，除非显式配置允许。
- 只对连接失败、连接超时、读取超时、部分 5xx 做重试。
- 支持 max_retry、retry_interval、退避策略。

第五步：实现多实例负载均衡

抽象 Endpoint：

- host
- port
- ssl
- weight
- status

每个 Endpoint 对应一个 HttpConnectionPool。

第一版负载均衡策略建议只做：

- RoundRobin
- Random

后续再做：

- WeightedRoundRobin
- LeastConnection
- 健康检查

第六步：实现限流保护

先做简单、可靠的并发数限制。

可以基于 FiberSemaphore 做：

- 全局最大并发
- 每个服务最大并发
- 每个 Endpoint 最大并发

后续再做 token bucket 来限制 QPS。

第七步：实现熔断降级

熔断要放在错误分类、重试、负载均衡之后做。

建议状态：

- CLOSED：正常请求
- OPEN：熔断打开，直接失败或走降级
- HALF_OPEN：允许少量探测请求

统计维度建议先按 Endpoint 做，再扩展到服务级别。

四、当前最适合立刻开始的任务

第一阶段已经完成：

- 修好连接池生命周期。
- 修好最大连接数控制。
- 增加连接池本地测试。
- 保证 cmake --build build 通过。
- 运行新测试验证连接复用、最大请求次数、最大存活时间、连接池满时等待/超时。

当前最建议继续做：

统一 HTTP 客户端超时模型

原因：

- HttpConnectionPool 已经具备基本可用的连接生命周期和容量控制。
- 当前 timeout_ms 同时承担等待连接超时、发送/接收超时等含义，语义还不够清晰。
- 后续错误分类、重试、负载均衡都需要明确区分 connect/send/recv/total timeout。

建议第二阶段目标：

- 新增 HttpRequestOptions 或类似参数对象。
- 明确 connect_timeout_ms、send_timeout_ms、recv_timeout_ms、total_timeout_ms。
- 让 HttpConnection 和 HttpConnectionPool 使用同一套超时参数。
- 保留当前 HttpConnectionPool 本地测试，并增加超时语义相关测试。
