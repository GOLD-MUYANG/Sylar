高并发 HTTP 能力建设路线

一、当前项目已经具备的基础

1. 高并发 HTTP 服务端基础

当前项目已经有 IOManager、Scheduler、Fiber、Hook、TcpServer、HttpServer 这一套基础。

IOManager 使用 epoll 管理 IO 事件，Hook 把 socket 读写接入协程调度，HttpServer 基于 TcpServer 完成 HTTP 请求解析、Servlet 分发和响应发送。

文档/压力测试.md 中已经记录过服务端压测结果：

- 短连接：ab -n 50000 -c 100，约 5448 QPS
- 长连接：ab -k -n 50000 -c 100，约 23650 QPS

所以“高并发 HTTP 服务端底座”已经有了。

2. HTTP 客户端基础

sylar/http/http_connection.h 中已经有 HttpConnection：

- DoGet()
- DoPost()
- DoRequest()
- recvResponse()
- sendRequest()

这些接口可以完成基础 HTTP 客户端请求，并且支持 HTTP 和 HTTPS。

3. HttpConnectionPool 第一阶段已修复

项目已经有 HttpConnectionPool：

- getConnection()
- doGet()
- doPost()
- doRequest()
- max_size
- max_alive_time
- max_request

并且已经完成第一阶段修复，可以作为后续 RPC/HTTP Client 的基础连接池继续建设。

本次已完成：

- 新建连接时设置 m_createTime。
- 修正 max_alive_time 判断，过期连接不再复用。
- 实现 m_maxSize 最大连接数限制。
- 连接池满时支持协程等待。
- 等待连接支持超时，超时后返回 POOL_GET_CONNECTION。
- 连接归还时统一判断是否还可复用，包括断开、过期、超过 max_request。
- 响应解析同步 Connection: close / keep-alive 状态，便于连接池决定是否复用连接。
- test_http_connection 使用 www.baidu.com 做连接池各项测试，覆盖真实外网目标。

仍需注意：

- 连续复用同一个 HTTP 连接发多次请求时，响应读取路径仍有进一步梳理空间，后续封装 HttpClient 时一并处理更合适。

4. 超时控制已有一部分

Hook 层已经有 connect/read/write 的协程超时机制。

HttpConnection 和 HttpConnectionPool 里也会设置 socket recv timeout。

但目前还不是完整的 HTTP 客户端超时模型。

后续最好统一成：

- connect_timeout：连接超时
- send_timeout：发送超时
- recv_timeout：读取响应超时
- total_timeout：一次请求总超时

5. Worker 和并发限制材料

项目里已经有 WorkerGroup 和 FiberSemaphore，可以作为“并发数限制”的基础。

第六步已经新增独立的 HttpConcurrencyLimiter，并接入 HttpLoadBalanceClient 请求链路，先完成 HTTP 客户端侧的并发数保护。

WorkerGroup 当前仍没有在实际 HTTP/RPC 请求链路中接入，而且构造函数里没有把 FiberSemaphore 初始化为 batch_size，直接使用会有卡住风险。后续如果要做 Worker 级别隔离，需要单独梳理 WorkerGroup 的边界。

6. 简单 worker 负载分摊

WorkerManager::get() 对同名 Scheduler 会随机返回一个，可以算是 worker 内部的简单负载分摊。

但这不是多实例 HTTP 服务负载均衡。

二、当前能力缺口和待增强点

1. 失败重试（基础能力已完成）

HTTP Client 层已经完成基础失败重试。

hook.cc 里的 retry 仍然只是系统调用层面对 EINTR/EAGAIN 的处理，不是业务请求重试；业务请求重试现在由 HttpClient 根据 HttpRetryOptions 和错误分类控制。

后续如果要扩展 RPC 语义、按接口配置重试策略、接入服务发现或熔断统计，可以继续在 HttpClient/RPC Client 之上增强。

2. 限流保护（基础能力已完成）

HTTP 多实例客户端链路已经完成第一版并发限流和 token bucket QPS 限制。

已完成：

- 独立新增 HttpConcurrencyLimiter，不把限流计数逻辑塞进负载均衡客户端。
- 支持全局最大并发。
- 支持每个服务最大并发。
- 支持每个 Endpoint 最大并发。
- 支持全局 QPS 限制。
- 支持每个服务 QPS 限制。
- 支持每个 Endpoint QPS 限制。
- HttpLoadBalanceClient 接入限流器，请求被限流时返回 RATE_LIMITED。
- 新增独立限流器测试，避免继续膨胀负载均衡测试文件。

仍需增强：

- 漏桶限流还没有实现。
- 当前是 tryAcquire 快速失败模型，暂不支持排队等待和等待超时。
- RPC 语义下的限流粒度，例如按接口、按调用方、按租户限流，还没有建设。

3. 熔断降级

当前没有 CircuitBreaker。

还缺少这些能力：

- 失败率统计
- 连续失败统计
- 熔断打开
- 半开探测
- 熔断恢复
- 降级回调或默认响应

4. 多实例负载均衡（基础能力已完成）

当前已经有面向多个服务实例的基础负载均衡能力。

已完成 Endpoint 抽象，并让每个 Endpoint 拥有自己的连接池。HttpLoadBalanceClient 会在请求时选择实例并转发请求。

后续还可以继续增强：

- 动态增删 Endpoint。
- 服务发现。
- Endpoint 级统计。
- 与熔断、限流、健康检查结果联动。

三、推荐建设顺序

推荐不要直接从熔断、限流、负载均衡开始。

这些能力都依赖稳定的客户端请求链路：

HTTP Client -> 连接池 -> 超时控制 -> 错误分类 -> 重试 -> 负载均衡 -> 限流/熔断

建议按下面顺序做。
