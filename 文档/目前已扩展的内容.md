高并发 HTTP 能力建设路线

一、当前结论

HTTP 客户端侧的主链路已经打通：

HttpConnectionPool -> HttpRequestOptions -> HttpClient -> retry -> HttpLoadBalanceClient
-> HttpConcurrencyLimiter -> HttpCircuitBreaker

目前已经完成：

- 连接池生命周期、容量、等待和超时控制。
- HTTP 客户端统一超时模型。
- 业务层 HttpClient 入口和错误分类。
- 失败重试策略。
- 多实例负载均衡。
- 并发数限制和 token bucket QPS 限制。
- Endpoint 维度熔断保护。

后续最适合继续做的是服务发现、RPC 语义、服务级熔断、降级回调和指标导出。

二、项目已有基础

服务端侧已经有 IOManager、Scheduler、Fiber、Hook、TcpServer、HttpServer。
`文档/压力测试.md` 中记录过服务端压测结果：

- 短连接：`ab -n 50000 -c 100`，约 5448 QPS。
- 长连接：`ab -k -n 50000 -c 100`，约 23650 QPS。

客户端侧已经有：

- `HttpConnection`：基础 HTTP/HTTPS 请求发送和响应读取。
- `HttpConnectionPool`：连接复用、容量控制、等待、归还和过期淘汰。
- Hook 层 socket 超时：connect/read/write 已接入协程调度。

三、集成入口

1. 单实例 HTTP 调用

优先使用 `sylar/http/http_client.h` 中的 `HttpClient`。

适用场景：

- 已知单个 host/port。
- 只需要连接池、超时、错误分类和重试。

相关能力：

- `HttpClient::Create()` 创建实例客户端。
- `HttpClient::request()/get()/post()` 复用实例内连接池。
- `HttpClient::Request()/Get()/Post()` 支持按 URL 直接请求。
- `HttpRequestOptions` 控制 connect/send/recv/total timeout。
- `HttpRetryOptions` 控制最大重试次数、间隔、退避策略和非幂等请求重试开关。

2. 多实例 HTTP 调用

优先使用 `sylar/http/http_load_balance_client.h` 中的 `HttpLoadBalanceClient`。

适用场景：

- 一个服务有多个 Endpoint。
- 需要负载均衡、健康检查、限流、熔断或 failover。

集成方式：

- 用 `HttpEndpoint::Create()` 创建每个实例。
- 每个 `HttpEndpoint` 内部持有自己的 `HttpConnectionPool`。
- 用 `HttpLoadBalanceClient::Create()` 传入 endpoint 列表、负载均衡策略、限流参数和熔断参数。
- 通过 `request()/get()/post()` 发起请求。
- 通过 `checkHealth()` 主动探测 endpoint，并更新 `UP/DOWN` 状态。

四、模块边界

1. 连接池

文件：

- `sylar/http/http_connection.h`
- `sylar/http/http_connection.cc`

职责：

- 管理单个连接和连接池。
- 控制连接最大数量、最大存活时间、最大复用次数。
- 连接池满时支持协程等待和等待超时。
- 根据连接状态、响应头和复用次数决定是否归还连接。

不负责：

- 不负责业务重试。
- 不负责多实例选择。
- 不负责限流和熔断。

2. 请求超时参数

文件：

- `sylar/http/http_request_options.h`
- `sylar/http/http_request_options.cc`

字段：

- `connect_timeout_ms`
- `send_timeout_ms`
- `recv_timeout_ms`
- `total_timeout_ms`

说明：

- 旧的 `timeout_ms` 入口仍保留，内部会转换成 `HttpRequestOptions`。
- connect/send/recv 阶段都会受 `total_timeout_ms` 剩余时间约束。

3. HttpClient

文件：

- `sylar/http/http_client.h`
- `sylar/http/http_client.cc`

职责：

- 作为业务层 HTTP 客户端入口。
- 持有并复用 `HttpConnectionPool`。
- 统一底层错误分类。
- 处理 HTTP 4xx/5xx 状态码映射。
- 执行失败重试策略。

重试规则：

- 默认不重试，`max_retry = 0`。
- 配置重试后，GET/HEAD 等幂等请求可以自动重试。
- POST/RPC 默认不自动重试，除非显式打开 `retry_non_idempotent`。
- 当前只对连接失败、连接超时、读取响应超时和部分 5xx 做重试。
- 5xx 中当前重试 500、502、503、504。
- 支持 `FIXED`、`LINEAR`、`EXPONENTIAL` 三种退避策略。

4. 多实例负载均衡

文件：

- `sylar/http/http_load_balance_client.h`
- `sylar/http/http_load_balance_client.cc`

职责：

- 维护多个 `HttpEndpoint`。
- 为每个 Endpoint 保留独立连接池。
- 选择可用 Endpoint 并转发请求。
- 集成限流器和熔断器。
- 在单次请求内遇到限流、熔断或可重试失败时尝试 failover 到其他 Endpoint。

Endpoint 字段：

- `host`
- `port`
- `ssl`
- `weight`
- `status`
- `active_request_count`

已支持策略：

- `ROUND_ROBIN`
- `RANDOM`
- `WEIGHTED_ROUND_ROBIN`
- `LEAST_CONNECTION`

健康检查：

- `checkHealth()` 通过 GET 探测 Endpoint。
- 探测成功则标记为 `UP`，失败则标记为 `DOWN`。
- 负载均衡选择时会跳过 `DOWN` Endpoint。

5. 限流保护

文件：

- `sylar/http/http_concurrency_limiter.h`
- `sylar/http/http_concurrency_limiter.cc`

职责：

- 独立维护并发数和 QPS token bucket。
- `HttpLoadBalanceClient` 只负责接入，不承载限流计数逻辑。

参数：

- `max_global_concurrency`
- `max_service_concurrency`
- `max_endpoint_concurrency`
- `max_global_qps`
- `max_service_qps`
- `max_endpoint_qps`

说明：

- `0` 表示对应维度不限制。
- 全局维度在进程内多个 limiter 实例之间共享。
- 服务维度只作用于当前 `HttpConcurrencyLimiter`。
- Endpoint 维度按 endpoint key 独立计数。
- 当前是 `tryAcquire()` 快速失败模型，不排队等待。
- 被限流时返回 `HttpResult::Error::RATE_LIMITED`。
- QPS 使用毫秒级 refill 的 token bucket，桶容量默认等于配置的 QPS。

6. 熔断保护

文件：

- `sylar/http/http_circuit_breaker.h`
- `sylar/http/http_circuit_breaker.cc`

职责：

- 独立维护 Endpoint 维度熔断状态机。
- 判断请求是否允许发出。
- 根据请求结果统计连续失败和窗口失败率。
- `HttpLoadBalanceClient` 只负责准入调用和结果上报。

状态：

- `CLOSED`：正常请求。
- `OPEN`：快速拒绝当前 Endpoint。
- `HALF_OPEN`：允许少量探测请求。

参数：

- `enabled`
- `consecutive_failure_threshold`
- `failure_rate_threshold`
- `failure_rate_min_request`
- `failure_window_size`
- `open_timeout_ms`
- `half_open_max_requests`

说明：

- 熔断默认关闭，避免旧调用行为变化。
- 需要调用方显式设置 `enabled = true`。
- `OPEN` 状态冷却到期后进入 `HALF_OPEN`。
- `HALF_OPEN` 探测成功后恢复 `CLOSED`，失败后重新打开。
- HTTP 5xx、连接失败、连接超时、发送失败、接收超时、响应解析失败计为失败。
- HTTP 4xx 不计为后端熔断失败。
- 被熔断时返回 `HttpResult::Error::CIRCUIT_OPEN`。

五、错误码和调用方语义

HTTP 客户端侧已经补充这些常用错误分类：

- `INVALID_URL`
- `INVALID_HOST`
- `CONNECT_FAIL`
- `CONNECT_TIMEOUT`
- `SEND_FAIL`
- `SEND_TIMEOUT`
- `RECV_TIMEOUT`
- `RESPONSE_PARSE_FAIL`
- `HTTP_STATUS_ERROR`
- `POOL_GET_CONNECTION`
- `RATE_LIMITED`
- `CIRCUIT_OPEN`

集成方建议按三类处理：

- 参数或地址错误：`INVALID_URL`、`INVALID_HOST`，直接修配置。
- 后端或网络失败：`CONNECT_*`、`SEND_*`、`RECV_TIMEOUT`、`RESPONSE_PARSE_FAIL`、`HTTP_STATUS_ERROR`，可结合重试、熔断和告警处理。
- 保护性拒绝：`POOL_GET_CONNECTION`、`RATE_LIMITED`、`CIRCUIT_OPEN`，应优先观察容量、限流阈值和后端健康状态。

七、下一步建议

1. 服务发现

目标：

- 支持动态增删 Endpoint。
- 让 `HttpLoadBalanceClient` 不再只能依赖静态 endpoint 列表。

建议边界：

- 服务发现单独成模块。
- 负载均衡客户端只消费发现结果，不直接实现注册中心协议。

2. 服务级熔断和降级

目标：

- 在 Endpoint 熔断之外，增加服务整体维度的保护。
- 支持降级回调或默认响应。

建议边界：

- `HttpCircuitBreaker` 继续负责状态机和统计。
- 降级响应不要塞进连接池或限流器。

3. RPC 语义

目标：

- 按接口、方法、调用方或租户设置超时、重试、限流和熔断。
- 为非幂等请求引入幂等键或业务级重试判断。

建议边界：

- RPC 参数对象和策略层应在 HttpClient 之上封装。
- 底层 HTTP 模块继续保持通用能力，不绑定具体业务协议。

4. 指标和观测

目标：

- 暴露 Endpoint 请求数、失败数、活跃请求数、限流次数、熔断状态和健康检查结果。
- 为后续压测、告警和自动扩缩容提供基础数据。

建议边界：

- 统计采集独立封装。
- 不把指标输出逻辑直接散进请求发送路径。
