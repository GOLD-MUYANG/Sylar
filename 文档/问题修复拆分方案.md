## F01：修复 `WorkerGroup` 无法投递任务

### 问题

`WorkerGroup` 构造时没有使用 `batch_size` 初始化 `FiberSemaphore`。信号量默认并发数为 `0`，所以第一次调用 `schedule()` 就会等待，后续任务没有机会执行，也无法归还名额。

此外，用户回调抛出异常时，`m_sem.notify()` 不会执行，会永久泄漏一个并发名额。

### 修改边界

- 修改 `sylar/worker.cc`：构造函数初始化 `m_sem(batch_size)`；用作用域守卫或等价的异常安全结构保证任务结束必归还名额。
- 新增 `tests/test_worker_group.cc`：只验证 `WorkerGroup` 的并发准入和异常后的名额回收。
- 修改 `CMakeLists.txt`：接入这个专用测试目标。

### 不做什么

- 不把 `WorkerGroup` 和 `WorkerManager` 合并。
- 不改变 `WorkerManager` 的命名 worker 配置模型。
- 不重写调度器或 `FiberSemaphore`。

### 验收

1. `batch_size = 1` 时，第一项任务可执行；不再首次阻塞。
2. 第一项任务结束后，第二项任务能继续执行。
3. 第一项任务抛异常后，后续任务仍能获得名额。
4. `cmake --build build --target test_worker_group` 和 `./bin/test_worker_group` 成功。

### 实现结果（2026-06-22）

- `WorkerGroup` 构造函数现在使用 `m_sem(batch_size)`，首个任务不再因默认 0 名额阻塞。
- `doWork()` 在回调异常时记录错误，并始终在末尾归还名额；不修改全局 `Fiber` 异常策略。
- 新增 `tests/test_worker_group.cc`，并在 CMake 接入 `test_worker_group`。

## F02：修复配置路径解析与静默空加载

### 问题

当前相对路径总是相对可执行文件目录解析。因此在仓库根目录执行：

```bash
./bin/sylar -s -c bin/conf
```

实际访问的是 `<仓库>/bin/bin/conf`；目录不存在时 `Config::LoadFromConfDir()` 又不会报错，服务会带默认配置继续运行。

### 修改边界

- 修改 `sylar/env.cc` / `sylar/env.h`：明确两类路径语义。
  - 用户显式传入 `-c <path>`：相对当前启动目录解析。
  - 未传 `-c`：默认使用可执行文件同级的 `conf/`。
- 修改 `sylar/config.cc` / `sylar/config.h`：加载目录不存在、不可读或未找到 YAML 时返回失败原因，不能静默成功。
- 修改 `sylar/application.cc`：配置加载失败时停止初始化并返回失败。
- 新增或启用专用配置测试，不依赖监听端口。
- 更新运行命令与配置路径说明。

### 不做什么

- 不实现配置热加载。
- 不改变 YAML 配置项的序列化格式。
- 不把配置目录强制改成绝对路径。

### 验收

1. 仓库根目录下 `./bin/sylar -s -c bin/conf` 能加载真实配置文件。
2. 未传 `-c` 时，仍能从 `./bin/conf` 找到默认配置。
3. `-c` 指向不存在目录时，初始化失败且进程返回非零状态。
4. 对合法配置目录的现有加载行为保持不变。

### 实现结果（2026-06-22）

- 新增启动工作目录记录：只有显式 `-c` 使用该目录作为相对路径基准；默认 `conf/` 仍位于可执行文件旁。
- `Config::LoadFromConfDir()` 现在返回成功状态，并区分目录不存在、不可读、没有 YAML、文件状态读取失败和 YAML 解析失败。
- `Application::init()` 在配置加载失败时返回失败，不再带默认配置继续启动。
- 新增 `tests/test_env_config.cc`，覆盖显式路径、默认路径、缺失目录和空目录。

## F03：修复连接池非法 URI 空指针崩溃

### 问题

`HttpConnectionPool::Create()` 在 `Uri::Create(uri)` 失败后只记录日志，随后仍访问 `turi->getHost()`，会直接空指针崩溃。

### 修改边界

- 修改 `sylar/http/http_connection.cc`：解析失败时返回空连接池指针，不再继续解引用。
- 在 `sylar/http/http_connection.h` 的工厂函数注释中明确：非法 URI 返回 `nullptr`。
- 在 `tests/test_http_client.cc` 或新增的 `tests/test_http_connection_pool.cc` 中加入回归测试；优先使用专用连接池测试文件，避免扩大无关测试。

### 不做什么

- 不修改 `Uri` 的语法规则。
- 不把连接池创建失败改成抛异常。

### 验收

1. 非法 URI 返回空连接池，不崩溃。
2. 合法 `http`、`https` URI 的 host、端口和 TLS 选择保持原行为。
3. 新增回归测试先失败、修复后通过。

### 实现结果（2026-06-22）

- `HttpConnectionPool::Create()` 在 `Uri::Create()` 失败时直接返回 `nullptr`，不再继续解引用。
- 工厂函数头文件注释明确非法 URI 的返回语义。
- 新增 `tests/test_http_connection_pool.cc` 和 CMake 目标。

### F04 实现结果（2026-06-22）

- `ModuleManager::initModule()` 仅在 `onLoad()` 返回成功后加入模块；失败模块会随着智能指针析构而销毁并关闭动态库。
- `Application::main()` 的退出路径统一停止 worker、卸载全部模块并删除 pid 文件。
- 新增独立动态插件与 `tests/test_module_lifecycle.cc`，覆盖 `onLoad` 与 `onUnload` 的计数和顺序。

### F05 实现结果（2026-06-22）

- `ModuleManager` 新增 `onServerReady()`、`onServerUp()`；全部模块回调均隔离异常和 `false` 返回，记录错误但不影响后续模块或连接清理。
- `Application::run_fiber()` 在所有服务 bind 后调用 ready，在所有服务 start 后调用 up。
- `TcpServer` 用 `SocketStream(owner=false)` 包装连接，在协议处理前后调用 connect/disconnect；模块不取得 socket 所有权。

### F06 实现结果（2026-06-22）

- `SSLSocket::ClientOptions` 提供 `server_name`、CA 文件/目录和 `verify_peer` 配置，默认开启 peer 校验。
- 客户端使用 `TLS_client_method()`，加载显式 CA 或系统默认 CA，设置 peer 校验、SNI 和 hostname 校验；握手后再次检查验证结果。
- `HttpConnectionPool` 创建 HTTPS socket 时把目标 host 作为 server name。
- 当前 sandbox 禁止创建 loopback socket，故证书成功、错误主机名和自签名拒绝的握手夹具应在具备本机网络权限的环境继续运行；默认 CTest 不依赖它。

### F07/F08/F09 实现结果（2026-06-22）

- `main.cc` 在 `Application::init()` 失败时返回 1；`run_fiber()` 的启动失败不再 `_exit(0)`，而是返回失败交给 `Application::main()` 清理。
- CTest 已注册 `worker_group`、`env_config`、`http_connection_pool`、`module_lifecycle`、`tls_client_options` 和 `startup_missing_config`，均标记为 `unit`。
- README 已删除重复“线程池”描述，改为清晰区分 `WorkerGroup` 与 `WorkerManager`，并同步插件/TLS 语义。

## F04：补齐插件加载与卸载生命周期

### 问题

动态库能被 `dlopen`，但 `Module::onLoad()` 没有调用；应用退出时也没有统一执行 `ModuleManager::delAll()`，导致 `onUnload()` 不保证执行。

### 修改边界

- 修改 `sylar/module.cc`：仅当 `onLoad()` 成功时才把模块加入管理器；失败时释放该模块及动态库。
- 修改 `sylar/application.cc`：在可控的应用退出路径中调用 `WorkerMgr::stop()` 与 `ModuleMgr::delAll()`。
- 新增测试模块和生命周期测试：用计数或可查询状态断言 `Create -> onLoad -> onUnload -> Destroy` 的顺序。

### 不做什么

- 不引入运行中热插拔、文件监控或模块版本迁移。
- 不在本项中处理客户端连接回调与服务 ready/up 回调（交给 F05）。

### 验收

1. `onLoad()` 被调用一次；若返回 `false`，模块不出现在管理器中。
2. 正常退出/显式卸载时，`onUnload()` 在动态库销毁前调用一次。
3. 无效模块符号、`onLoad()` 失败都不会泄漏动态库句柄。

## F05：补齐插件服务与连接生命周期

### 问题

`onServerReady()`、`onServerUp()` 没有调用；`ModuleManager::onConnect/onDisconnect()` 虽有实现，却没有接入 TCP/HTTP 服务端的连接处理路径。

### 修改边界

- 修改 `sylar/application.cc`：所有服务绑定成功后调用 `onServerReady()`；所有服务启动后调用 `onServerUp()`。
- 修改 `sylar/tcp_server.cc` 或明确的 HTTP 服务入口：在连接处理前后触发模块连接回调，并传递拥有 socket 的 `Stream` 包装对象。
- 新增专用测试模块，验证 ready、up、connect、disconnect 的调用顺序和次数。

### 不做什么

- 不允许模块直接篡改 HTTP Servlet 路由表；这属于另一种扩展点。
- 不定义复杂的回调失败重试策略；第一版仅记录失败并保持服务自身的连接清理语义。

### 依赖

- 依赖 F04，确保模块的 load/unload 基础语义可靠。

### 验收

1. 一个成功启动的服务会按 `onLoad -> onServerReady -> onServerUp` 顺序通知模块。
2. 一条连接会对应一次 `onConnect` 和一次 `onDisconnect`。
3. 模块回调异常或返回失败不会导致 socket/会话资源泄漏。

## F06：让 HTTPS 客户端进行身份校验

### 问题

客户端仅调用 `SSL_connect()`，没有加载可信 CA、没有要求校验证书、没有校验主机名，也没有设置 SNI。因此 HTTPS 连接可被错误证书或中间人证书接受。

### 修改边界

- 修改 `sylar/socket.h` / `sylar/socket.cc`：为客户端 TLS 增加 server name、CA 路径和是否校验 peer 的明确配置；安全默认值为校验证书和主机名。
- 修改 `sylar/http/http_connection.cc`：连接池把目标 host 传给 TLS socket，用于 SNI 与主机名验证。
- 新增独立 TLS 测试夹具：验证受信任证书成功、错误主机名或不受信任证书失败。

### 不做什么

- 不实现客户端证书（mTLS）。
- 不实现自定义证书钉扎策略。
- 不修改服务端证书加载格式（仍使用 PEM）。

### 验收

1. 合法 CA 和匹配主机名的 HTTPS 请求成功。
2. 自签名且未显式信任、主机名不匹配的连接失败并返回清晰错误。
3. ClientHello 带有目标主机的 SNI。

## F07：修复启动失败被报告为成功

### 问题

配置地址非法、worker 不存在、bind 失败时，`Application::run_fiber()` 使用 `_exit(0)` 退出。对 shell、守护进程和部署系统而言，这是“成功退出”；同时跳过了析构和清理。

SSL 证书加载失败也只写日志，随后仍启动监听，产生“进程存在但 TLS 服务不可用”的假健康状态。

### 修改边界

- 修改 `sylar/application.cc`：把启动失败沿正常调用链返回；进程使用非零退出码。
- 证书加载失败时停止对应服务启动，而非继续运行。
- 为启动阶段补一个可观察的失败状态，避免异步 `run_fiber()` 失败被调度器吞掉。
- 新增配置错误、未知 worker、无效证书三类启动失败测试；测试使用临时配置和临时目录，不复用 `bin/conf`。

### 不做什么

- 不在本项实现完整的优雅停机信号处理。
- 不改变成功启动时的 HTTP 监听和 worker 分配策略。

### 验收

1. 非法地址、未知 worker、bind 失败、证书失败均返回非零状态。
2. 失败后不遗留 pid 文件、监听 socket 或未停止的 worker。
3. 成功配置仍能进入原有启动路径。

## F08：建立 CTest 测试发现和分层运行入口

### 问题

当前项目没有 `enable_testing()` 和 `add_test()`，执行 `ctest --test-dir build` 会显示 `No tests were found`。同时，CMake 中大量基础测试被注释，回归验证依赖人工记忆可执行文件名称。

### 修改边界

- 修改 `CMakeLists.txt`：启用 CTest，并注册已经稳定、无需外网、无需受限 loopback 权限的测试。
- 为受环境限制的 socket/外网测试添加标签，例如 `integration`、`network`；默认单元测试不依赖它们。
- 逐个恢复基础测试目标前先确认它们可编译、可重复运行；不能因为“源码存在”就批量取消注释。
- 文档写明三层命令：构建、默认单元测试、可选集成测试。

### 不做什么

- 不引入 GoogleTest、Catch2 等新测试框架。
- 不承诺一次性恢复所有历史测试。

### 验收

1. `ctest --test-dir build --output-on-failure` 至少能发现并运行稳定单元测试。
2. 默认测试不访问外网，也不要求本环境允许创建 loopback socket。
3. 集成测试可显式按标签运行，并在缺少环境能力时给出明确说明。

- 明确插件生命周期实际支持的回调范围。
- 明确 HTTPS 的默认校验语义和配置入口。
- 删除“已经实现”但没有可验证入口的夸张描述。

### 不做什么

- 不把 README 改写成完整教程。
- 不调整你现有的个人项目叙事风格；仅校准重复和事实准确性。
